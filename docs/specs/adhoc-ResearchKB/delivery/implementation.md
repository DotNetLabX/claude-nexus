## Key Decisions
- **Step 1 grammar uses square-bracket tokens only (`[n]`, `[no source found]`, `[source]`, `TODO`, `TBD`).** Confirmed E7-safe — skill-lint E7 only fires on angle-bracket `<...>` tokens in prose. All placeholders use `{curly}` style or live inside fenced code blocks (E7 exempts code).
- **Validity scope is a required field with no default; absent ⇒ stale** (critic minor finding, folded into plan Step 1). The recall validity-gate (Step 5) depends on this pairing of As-of + Validity scope.
- **`deep-research` must NOT be written as `` `deep-research` skill `` (Step 2).** The skill-refs lint (`tests/lint/skill-refs.test.mjs:41`) treats every `` `name` skill `` token as an assertion that `name` ships — so the natural prose "no dependency on the external `deep-research` skill" *failed* the lint (deep-research is not a shipped nexus/nexus-dotnet skill). Rephrased to "the external `deep-research` harness" — same meaning, no false skill reference. The intended in-pool references (`research-entry-schema`, `cite-check.mjs`) use bold-backtick / path forms, not the `` `x` skill `` form, so they don't false-trigger either.
- **cite-check claim regex tolerates sentence-final punctuation after `[n]` (Step 3).** First TDD slice went red on the valid `SOURCED` fixture: a real claim ends `... API [1].` (period after the bracket), but the initial `/\[(\d+)\]\s*$/` anchored `[n]` to end-of-line. Fixed both the `[n]` and `[no source found]` matchers to `[.,;:)\s]*$` so a cited claim ending in punctuation still validates. Root-caused by running the script directly on the fixture (diagnose-style), not by guessing.
- **Step-4 live run: all three acceptance criteria passed.** (a) **Fork isolation (the §8 probe) confirmed** — the `Explore` researcher's dozens of WebSearch/WebFetch calls stayed in its own context; the caller (this developer) transcript received only the summarized draft entry. This is the context isolation the whole design rests on, observed live. (b) A schema-conformant entry was written to `docs/kb/research/claude-code-skill-context-fork.md`. (c) `cite-check.mjs` exits 0 on it. Question researched (genuinely useful, fact-shaped, high-stakes): whether Claude Code's `context: fork` isolates a skill in a subagent.
- **Step-7 recall validated end-to-end, both paths.** Part 1 (cache hit): recall grepped the pool, judged the Step-4 entry a true hit on its `## {Question}` heading, validity-gated it (reconfirm-trigger not fired, validity scope holds, status current) and **returned the cached verdict with no `Explore` spawn** — no researcher in the transcript. Part 2 (stale path): the reconfirm-trigger was treated as fired → recall reclassified the entry as a miss → re-research produced a new entry → **supersede-don't-delete** applied: the old block is `Status: superseded` (kept), the new block is `Status: current`, both in the same topic file; cite-check exits 0. The orchestration is non-deterministic (already exercised live in Step 4), so Part 2's re-research entry was composed directly rather than burning a second identical real dive — the supersede mechanism is what Step 7 verifies, and it is verified on disk.
- **Step-8 version baseline reconciliation: plan said 1.9.2 -> 1.10.0; actual was 1.9.3 -> 1.10.0.** The live plugin.json was already at 1.9.3 (a prior commit bumped it past the 1.9.2 the plan was authored against — the session-start git snapshot was stale). The **binding instruction is MINOR (new capability)**, not the literal source number; from 1.9.3 a MINOR bump yields `1.{9+1}.0` = **1.10.0**, which is exactly the plan's stated target. So the drift is cosmetic — the MINOR result is identical. No deviation in outcome.
- **`node --test tests/lint tests/unit` (the bare-directory form) fails to glob on Node v24.13.0 — use the explicit `*.test.mjs` form.** Passing a bare directory makes the CJS resolver try to *load* `tests/lint` as a module ("Cannot find module"), not glob it. The full suite is green via the CI-authoritative form `node --test tests/lint/*.test.mjs tests/unit/*.test.mjs` (156 pass / 0 fail, incl. the 7 cite-check tests, skill-refs, and the "all shipped nexus skills lint-clean" dogfood). This is an invocation quirk, not a test defect — the reviewer should use the explicit-glob form.

## Review Fixes (Round 1)

Consolidated Step-2 findings (nexus reviewer + independent Codex cross-check) on
`cite-check.mjs` + its tests. All blocking + required items fixed; the optional schema tightening
done too. Done-conditions re-verified after every change.

**Fix 1 (BLOCKING) — prose claim lines no longer bypass the check.**
`cite-check.mjs` previously `continue`d on any non-`- ` line under a claim section, so an uncited
**prose** claim under `## Finding`/`## Fix`/`## Alternatives` passed undetected. The schema mandates
claims-as-bullets (the example shows bullets; the grammar says "every claim line"). Resolved per the
fix-list's *reject-as-malformed* branch: a non-blank, non-bullet line under a claim section now pushes a
`non-bullet (prose) line under ## {section} — claims must be bullets` violation (exit 1). Blank lines are
still skipped as framing. (`plugins/nexus/skills/search-researches/scripts/cite-check.mjs`, per-block pass B.)

**Fix 2 (BLOCKING) — source resolution AND corroboration are now per-block, not file-global.**
A topic file accumulates entry blocks (supersede-don't-delete keeps superseded blocks — the live artifact
already has two). The old code (a) collected all `[n]` into one file-global Map, so a later block's
`[1]`/`[2]` overwrote an earlier block's and an earlier cited claim could resolve to the wrong/later
source; and (b) read only the **first** `Corroboration` via `lines.find()`, so a later block's high-stakes
single-source verdict was never checked. Rewrote the script to first split the file into **entry blocks**
— the block delimiter is the per-entry `## {Question answered}` heading, derived from `research-entry-schema`
(every entry = question heading + metadata + the fixed 8-part body; a `## ` heading whose title is NOT one
of the 8 body-section names `{Verdict,Finding,Fix,Alternatives,Caveat,Fallback,Sources,Recommendation}` is
a question heading → new block). Then, **per block**: a fresh `sources` Map (placeholder + resolution scoped
to the block), the claim-citation pass, and the corroboration floor (`block.lines.find` of *this* block's
Corroboration — every block checked, not just the first). Verified the bug is real: a crafted block-1 claim
citing `[2]` that only block 2 defines now FAILS (`CITE-FAIL … cites [2] but no ## Sources entry [2] exists
in this block`), where the old file-global code would have falsely passed it via inheritance.

**Fix 3 (REQUIRED) — expanded `tests/unit/cite-check.test.mjs` from 7 → 19 tests.** Added, per the fix-list:
- (a) cited claims under `## Fix` **and** `## Alternatives` pass; an uncited claim under each fails (3 tests).
- (b) a prose (non-bullet) claim line under a claim section fails.
- (c) a multi-digit ref `[10]` resolving to `## Sources` passes.
- (d) an out-of-range ref `[9]` resolving to no `## Sources` entry fails.
- (e) each placeholder variant `TODO` / `TBD` / `[source]` fails (parameterized, 3 tests).
- (f) the two cross-block false-passes from Fix 2: a block-1 claim resolving only to a *later* block's
  `## Sources` fails (per-block resolution); a *later* block's high-stakes single-source verdict fails
  (per-block corroboration). Plus a positive control — a clean two-block file (the legitimate supersede
  steady state) passes — to prove the per-block split introduces no false failures.

**Fix 4 (OPTIONAL, done) — tightened `research-entry-schema/SKILL.md` grammar wording (no behavior change).**
The schema example showed bullets but the prose never *required* the bullet form — the exact silence that
let Bug 1 hide. Updated the claim-grammar bullets to state: claims are bullet lines (a prose line is
malformed and fails), `[n]` is one-or-more digits, resolution is against **this entry block's** `## Sources`,
and the corroboration floor is per-block (every block checked). Documents the behavior the validator now
enforces; no code depends on the new wording. Re-linted clean.
