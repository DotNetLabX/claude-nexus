# adhoc-MvcSurvivorReport — Lessons

## Architect Lessons

- **A "pure classifier" over mutant metadata is a trap when the tags are source-semantic.** The plan specified a pure fn mapping `{line, mutatorName, replacement}` → one of 5 survivor tags. Three tags (`dead-code`, `masked`, `equivalent-format`) are cross-procedural / whole-source properties the orchestrator (no fs) cannot derive — and `mutatorName` carries the mutation *kind*, not the source *text*, so even `equivalent-logging` isn't safely derivable from a line number. The code-grounded critic caught it as CRITICAL. **Lesson:** before specifying a pure orchestrator fn, check that EVERY output value is derivable from the declared input alone; a property that needs the source must be assigned by an agent with source access, with the orchestrator only recording the verdict. Phantom input fields (`kbFlagged`, which nothing in the pipeline produces) are the tell that the design is reaching for context it doesn't have.

- **An anti-fake-green invariant must not force the very fabrication it guards against.** The plan's cross-check `mutants.length === found` would force the runner to synthesize 54 "Killed" entries the survivors-only XML cannot supply — fabrication, inside the honesty gate. The correct invariant cross-checks what the tool actually emits (`summary.undetected === survivor-array.length`) and rescpres the floor from the summary numbers, never from counting a fabricated array. **Lesson:** when hardening a gate, the cross-check must be expressible from the tool's *real* output shape; if satisfying it requires inventing data, the invariant is wrong, not the tool.

- **Re-grounding a fix plan against a SHIPPED contract beats re-grounding against the proposal.** The plan claimed "the reference matches the new skill contract" but specified array-counting while the just-shipped adapter prescribed summary-arithmetic. The two drifted because the plan was written against the feedback doc, not the solo-edited skill. **Lesson:** when a plan's job is to make a reference impl match a contract, the binding source is the *current shipped contract file*, re-read after any concurrent fix lands — not the originating proposal.

- **Code-grounded review is non-optional for harness/skill passes — and it paid out exactly as the rule predicts.** A doc-only critic would have rubber-stamped the taxonomy routing (which was fine) and missed the CRITICAL + the two HIGH scoring defects, all of which were visible only by reading `cover-flutter.workflow.js` (the no-fs constraint at :35, the survivors-only schema at :242-246, the loop control flow at :387-404) and running the live suite (350 tests / 4 pre-existing failures, not the plan's claimed 315). Mandating code-grounded review for this pass was correct.

- **A stale baseline number in an acceptance line is a developer trap.** "315 tests, 0 fail" was wrong (350 / 4 pre-existing failures) and would have blocked or scope-crept the developer. **Lesson:** acceptance that gates on a suite count must state the count verified *this round* and scope the green gate to the changed file, calling out unrelated pre-existing failures explicitly.
