# adhoc-MvcSurvivorReport — Lessons

## Architect Lessons

- **A "pure classifier" over mutant metadata is a trap when the tags are source-semantic.** The plan specified a pure fn mapping `{line, mutatorName, replacement}` → one of 5 survivor tags. Three tags (`dead-code`, `masked`, `equivalent-format`) are cross-procedural / whole-source properties the orchestrator (no fs) cannot derive — and `mutatorName` carries the mutation *kind*, not the source *text*, so even `equivalent-logging` isn't safely derivable from a line number. The code-grounded critic caught it as CRITICAL. **Lesson:** before specifying a pure orchestrator fn, check that EVERY output value is derivable from the declared input alone; a property that needs the source must be assigned by an agent with source access, with the orchestrator only recording the verdict. Phantom input fields (`kbFlagged`, which nothing in the pipeline produces) are the tell that the design is reaching for context it doesn't have.

- **An anti-fake-green invariant must not force the very fabrication it guards against.** The plan's cross-check `mutants.length === found` would force the runner to synthesize 54 "Killed" entries the survivors-only XML cannot supply — fabrication, inside the honesty gate. The correct invariant cross-checks what the tool actually emits (`summary.undetected === survivor-array.length`) and rescpres the floor from the summary numbers, never from counting a fabricated array. **Lesson:** when hardening a gate, the cross-check must be expressible from the tool's *real* output shape; if satisfying it requires inventing data, the invariant is wrong, not the tool.

- **Re-grounding a fix plan against a SHIPPED contract beats re-grounding against the proposal.** The plan claimed "the reference matches the new skill contract" but specified array-counting while the just-shipped adapter prescribed summary-arithmetic. The two drifted because the plan was written against the feedback doc, not the solo-edited skill. **Lesson:** when a plan's job is to make a reference impl match a contract, the binding source is the *current shipped contract file*, re-read after any concurrent fix lands — not the originating proposal.

- **Code-grounded review is non-optional for harness/skill passes — and it paid out exactly as the rule predicts.** A doc-only critic would have rubber-stamped the taxonomy routing (which was fine) and missed the CRITICAL + the two HIGH scoring defects, all of which were visible only by reading `cover-flutter.workflow.js` (the no-fs constraint at :35, the survivors-only schema at :242-246, the loop control flow at :387-404) and running the live suite (350 tests / 4 pre-existing failures, not the plan's claimed 315). Mandating code-grounded review for this pass was correct.

- **A stale baseline number in an acceptance line is a developer trap.** "315 tests, 0 fail" was wrong (350 / 4 pre-existing failures) and would have blocked or scope-crept the developer. **Lesson:** acceptance that gates on a suite count must state the count verified *this round* and scope the green gate to the changed file, calling out unrelated pre-existing failures explicitly.

- **A "do X to BOTH A and B" plan instruction can be a verified no-op for one of them — pre-resolve which is the real target.** F3 said add `mutationSummary` to both the 9e and 9f cover-shaped mocks. 9e needed it (runs the real cover-flutter); 9f did NOT — loop-flutter never reads `coverResult.mutationSummary` (the new gate lives inside cover-flutter, which 9f mocks), so the field would be dead data. The developer correctly resolved the hedge against source; at done-check it reads as `Deviated (valid reason)`, not Missing. **Lesson:** when a plan step touches two test mocks "so they stay green under a new gate," trace where the gate actually fires before mandating the edit on both — a mock of the sub-workflow that hosts the gate needs no change. State the data-flow (which file reads the field) in the step so the no-op is pre-sanctioned instead of escalated.

- **A blocking full-suite verify gate is not automatically a Fail — separate introduced from pre-existing by identity, not count.** The session gate failed on the full suite + selfcheck, but the same 4 failures (gen-omni ×3 on synthetic sandboxes, nexus-cpp ×1 CHANGELOG bracket lint) were recorded at the zero-change analyze phase and are structurally independent of every touched file. Delta 350/4 → 358/4 = +8 tests all passing, 0 new failures. **Lesson:** at done-check, re-run the suite, list the failing tests by name, and confirm each is pre-existing AND independent of the changed files (synthetic-sandbox tests can't see a version bump; an untouched plugin's lint can't regress) — a scoped green gate on the changed file plus an identity-matched baseline is the PASS basis, not the raw red/green of the whole-tree gate.

- **A dual-purpose array is the seam where a "stays in the list" plan note silently violates a behavioral contract.** `reachableSurvivors` served BOTH the report-render list AND the next-iteration cover-feedback list. Plan Step 4 said "equivalent-logging stays in reachableSurvivors so loop-flutter can render it" — true for the report, but the same array is the loop-feedback source (cover-flutter:460), so the pre-tag also re-fed known-equivalent mutants into the next Cover prompt, violating the shipped SKILL.md:68 "only REAL-gap drives another iteration." The nexus reviewer APPROVED it (blessed the report-layer design); the Codex source-grounded cross-check caught the feedback leak. **Lesson:** when a plan keeps an item "in" a collection for one consumer, enumerate EVERY consumer of that collection — if a list is both rendered and fed back, a membership note for the render path is a behavioral change for the feedback path. Done-checks/reviews must trace the array's second use, not just the one the plan names.

- **A schema/merge keyed by a non-unique field collapses distinct records — verify the key is 1:1 with the entity.** The classify contract keyed verdicts by source `line`; regex mutation emits multiple survivors per line, so `Map.set(c.line)` / `.get(s.line)` last-write-wins collapsed them and both rendered the same tag — though the prompt said "one entry per survivor." **Lesson:** when a plan says "one X per entity", confirm the join key is unique per entity (add an index/id when the natural field isn't 1:1) — a per-line key for a per-survivor contract is a silent misclassification, invisible to a happy-path test that never puts two records on one key.

- **Adjudicating a reviewer-vs-Codex split: re-verify every external citation, then rule by mechanism — a green suite that never exercises the path is not evidence the path is correct.** Codex NO-GO vs nexus APPROVE: I re-read each cited line (all accurate — not a fabrication this time, unlike the prior Codex-schema case), found the reviewer's "merge is correct" positive observations were the exact misses, and ruled F1/F2 REAL (HIGH) over the APPROVE. The 41/41 green suite did not contradict the findings — no slice covered same-line collision or feedback filtering. **Lesson:** "the suite is green" and "Codex couldn't run the suite" are both non-arguments when the findings are behavioral paths the suite never exercises — adjudicate by tracing the mechanism in live source, and split a finding (INTENDED denominator / REAL feedback) rather than rubber-stamping it whole.

## Developer Lessons

- **Run the scoped contract suite during Phase-1 analysis, not just at Phase-2 verify.** Executing `node --test tests/unit/workflow-contract.test.mjs` (read-only) during analysis confirmed the plan's F7 claim (33/33 green) first-hand and de-risked the two medium-confidence steps (3 + 4) before any code: I now know the exact green starting point and that slices 9e/9f are the only existing flutter-touching tests. **Lesson:** when a plan's acceptance pins a suite baseline, verify it live in Phase 1 — it costs one read-only run and turns a plan *claim* into a measured fact, so Phase-2 deviations are unambiguous.
- **Plan line anchors held up 1:1 against live source — a sign of a code-grounded plan.** Every anchor (`cover-flutter.workflow.js:95` reachableSurvivors push, `:259` schema `required`, `:379-384` ratchet break; `loop-flutter:198-201,227`; test `:690-702` 9e) matched exactly. The rev-2 (critic-folded) plan's anchors were trustworthy enough to navigate by. No new convention gap found.
- **`deepStrictEqual` fails on arrays built inside a `vm` context — assert membership, not identity.** The contract suite runs each workflow in `vm.createContext`, so any object the script returns (or an `opts.schema` it passes to the mock `agent`) belongs to the sandbox realm. `assert.deepEqual([...vmArray], [...outerArray])` (which `node:assert/strict` makes `deepStrictEqual`) trips the cross-realm prototype check even when contents are identical — the failure prints actual === expected, which is the tell. Use `Array.isArray(x) && x.length === N` + `x.includes(v)` (both cross-realm safe) instead. A stronger assertion anyway (I also verified `equivalent-logging` is excluded from the agent enum). **Lesson:** for vm-sandbox returns, prefer membership/scalar assertions over deep-equality on objects.
- **`bump-plugin.mjs` bumps EVERY changed plugin per invocation — run it ONCE after all plugin edits, never per-step.** It classifies the whole working tree vs HEAD. Bumping after Step 1 then again after Step 2 would re-detect the still-changed Step-1 skill and double-bump it. Land all plugin-file edits first, then a single run produces the correct one-bump-per-plugin set (`--dry-run` to confirm the targets). **Lesson:** a multi-plugin change is one bump event, not N.
- **A survivors-only contract change rewrites the test FIXTURE shape, not just adds a field.** F3 read as "add `mutationSummary` to 9e", but the deeper change (mutants array → survivors-only) meant 9e's pre-existing `Killed` entry had to be REMOVED too — a `Killed` left in a now-survivors-only array would be mis-scored as a survivor. When a contract flips what an array MEANS, audit every fixture for entries that silently violate the new meaning, not just for the missing new field.
- **Keying a per-item verdict by a NON-unique field (source line) is a last-verdict-wins data-loss bug.** F2: two survivors on one source line collided in a `Map` keyed by line — the second verdict silently overwrote the first. Fix: give the agent a stable per-item `index`, have it echo the index back, and key by index. The robust merge avoids object identity across the `vm`/composition boundary by relying on ORDER instead: `toClassify = survivors.filter(!tag)` preserves order, so walking `survivors` with a running counter recovers each item's index without a reference-identity `Map`. **Lesson:** when a model echoes back a list, key its responses by an explicit index you assigned, never by a content field that can repeat.
- **In a fix-cycle on an UNCOMMITTED feature, `bump-plugin.mjs --dry-run` proposes `current+1` — that is NOT a mandate to re-bump.** The tool reads `cur` from the WORKING TREE (already at this feature's bump, e.g. 1.18.6) and always offers `cur+1` for any plugin file dirty-vs-HEAD. A same-feature edit (here a reviewer-finding SKILL.md fix) is covered by the existing uncommitted bump — applying the proposal would wrongly create a 1.18.7. **Lesson:** the dry-run's left-hand version (the `cur`) is the bump that will ship; only apply a new bump when `cur` equals the COMMITTED/HEAD version. Strengthens "one bump event per change, not per-step" — a fix-cycle is part of the SAME change event.

## Reviewer Lessons

- **Algebraic reduction is the fastest check for scoring-formula correctness.** For `mutationFloor`'s formula, substituting `reachableSurvivors.length = undetected - exclusions` and `reachableDenominator = (found - notCovered) - exclusions` reduces `killed` to `found - notCovered - undetected = killed_by_tests + timeouts` — a single line that proves correctness and explains why timeouts count on the killed side. This is faster and more conclusive than tracing the loops. Use algebraic reduction first for any arithmetic gate formula.
- **A targeted 5-prediction pre-commitment list sharpens a Step 3–4 focused review.** Writing down the 5 most likely failure modes (zero-denominator, break-vs-gate, purity, enum alignment, required-field) before reading the code made it easy to route each read to the right place and confirm or refute each claim. All 5 turned out correct; the pre-commitment list also revealed what NOT to spend time on (the doc steps 1-2 were genuinely low-risk as predicted).
- **A Map that keys agent verdicts by source-line is not stable when two survivors share a line.** The first pass confirmed the `classifyByLine.set(c.line, c)` merge was "correct" without asking whether the key was a unique identifier. Regex mutators routinely emit multiple survivors per line (e.g., two operators on one compound expression) — a line key lets the second verdict silently overwrite the first (last-verdict-wins). The fix is a per-survivor index the agent echoes back. **Rule:** when reviewing any Map or dict that keys classifier output, verify the key uniquely identifies one survivor in ALL input shapes, not just the test fixture.
- **The re-feed path is a distinct consumer of `reachableSurvivors` that needs its own trace.** The first pass verified that `toClassify = survivors.filter((s) => !s.tag)` withholds pre-tagged survivors from the classify agent — but `survivingMutants = reachableSurvivors` (the old re-feed line) was a DIFFERENT consumer of the same structure at a different point in the loop, and it was not traced. A data structure read at multiple call sites (classify agent + iteration re-feed) requires a separate review of each consumer. Missing one consumer left F1 undetected.
