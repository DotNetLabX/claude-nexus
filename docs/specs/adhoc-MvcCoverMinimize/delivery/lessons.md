# Lessons — adhoc-MvcCoverMinimize

## Architect Lessons

- **In the MVC harness the orchestrator has no filesystem — confirm / apply / restore are agent ops, never orchestrator ops.** Rev 1 put the confirm re-mutation and the test-file edits on the "pure/deterministic" orchestrator. That is structurally impossible (`mine-verify-cover/SKILL.md:30` — agents do all I/O) and produced a fake-green confirm (a pure recompute over the stale kill-map can never regress). **Apply:** in any MVC/harness plan, mark every I/O step's owning agent explicitly (minimize agent reasons; write-agent edits; runner re-runs; orchestrator only routes + decides) — mirror the live classify-survivors split, don't reinvent it.

- **A fail-closed confirm is only real if it RE-RUNS fresh and compares EXACT counts.** Two independent ways rev 1's confirm could pass while coverage dropped: (a) recomputing over the pre-minimize numbers instead of a fresh re-gate; (b) comparing the rounded `scorePct` (a one-mutant drop on a large denominator rounds away). **Apply:** confirm = a fresh runner call on the reduced suite + an exact `killed`-count compare + restore-on-any-drop. A confirm with no fresh-run assertion in its test is vacuous.

- **Per-test mutant attribution is fallible reasoning that NO tool can verify — name it as a hypothesis and make the confirm the only safety net.** `mutation_test` reports aggregate survivors, never which *test* killed which mutant. A contract test that mocks a per-test kill-map tests fabricated data. **Apply:** ground minimize tests on the real aggregate output shape (`mutationSummary`); the re-gate is the verifier, not the attribution.

- **Code-grounded review earned its mandate here — a doc-only critic would have shipped the defects.** Both the code-grounded critic (HIGH-1) and Codex (CRITICAL + 2 HIGH) converged on the same seam, and every blocker was visible only by reading live source (the no-filesystem contract, the harness output shape, the rounded-vs-exact compare). Confirms the ADR done-check rule: for a shared/external-artifact (skill) change, code-grounded review is the load-bearing gate, not a plan-vs-spec diff.

- **Don't read a dirty working tree's version as the bump baseline.** Working-tree `plugin.json` showed 1.18.7, but committed HEAD was 1.18.6 — the 1.18.7 was a *concurrent* feature's uncommitted bump. A naive `bump-plugin` run would have bundled the foreign feature. **Apply:** bump from committed HEAD at implementation time, scope staging to this feature's files, recheck branch/log before commit (the recheck-branch-under-concurrent-run discipline).

- **Done-check (Step 1): when a step's whole safety argument rests on "copied verbatim, KEEP IN SYNC", byte-diff the copy — don't accept that the copy exists.** The Minimize confirm is apples-to-apples only if `loop-flutter`'s `mutationFloor` + its deps (`mutantLine`/`VOID_CALL_REMOVAL_RE`/`pretagEquivalentLogging`) produce the IDENTICAL `killed` count the pre-minimize gate produced in `cover-flutter`. I verified the executable bodies are byte-identical (comments diverge, logic doesn't) and that both call sites pass the SAME `expectedSurvivorLines`/`equivalentLoggingLines` args — that, not the presence of a confirm agent, is what makes `killedBefore` vs `killedAfter` a real comparison. **Apply:** for any "verbatim-copy-with-keep-in-sync" step, the done-check reads both copies and diffs the load-bearing function, and confirms the two invocations use matching inputs.

- **Done-check (Step 1): a parenthetical file in a plan's Files line is a reuse pointer, not a mandatory diff target — resolve it as Deviated-valid, not Missing.** Plan Step 3 named `cover-flutter.workflow.js (reuse the existing runner-agent re-gate path…)`; the developer copied its helpers into `loop-flutter` and left `cover-flutter` untouched. That is not a skipped step: the Workflow runtime forbids cross-file import, and re-composing `cover-flutter` via `workflow()` would re-run a Cover agent that regenerates/discards the just-reduced suite (defeating the confirm). The plan's acceptance was about `loop-flutter`'s behavior, which is fully met. **Apply:** when a Files-line parenthetical describes a *source to reuse*, the done-check tests the acceptance (the behavior), not a literal diff to the cited file.

## Developer Lessons

- **"Reuse the existing X path" in a Workflow-script plan step means verbatim-copy-with-keep-in-sync, not a cross-file call or a nested `workflow()` composition — and this is now a recurring pattern, not a one-off.** `cover-flutter.workflow.js` already copies its gate battery from `cover.workflow.js` verbatim (documented "KEEP IN SYNC"); this feature's confirm re-gate needed the SAME treatment for `mutationFloor` + its helpers, copied from `cover-flutter.workflow.js` into `loop-flutter.workflow.js`. A tempting alternative — composing `cover-flutter.workflow.js` again via `workflow()` for the confirm — is actively wrong here: that sub-workflow's iteration 1 always spawns a Cover agent first, which would silently regenerate/discard the just-reduced test file the confirm is supposed to be gating. **Apply:** when a plan step says "reuse the existing runner/gate path" inside a Workflow script and no shared module exists to import from (the runtime forbids static import), read it as "copy the pattern, document why, keep in sync" — never as license to nest a sub-workflow whose own control flow (e.g. always-write-first) would defeat the intent.
- **A test-helper's `Math` mock built as `{ random: () => throw }` alone silently degrades to `Math.round is not a function` the moment new production code under test calls `Math.round`.** The shared `runInSandbox` helper in `workflow-contract.test.mjs` already builds a full proxy (every `Math` method except `random`), but several ad-hoc inline sandboxes elsewhere in the same file copied only the minimal `{ random: ... }` stub — which was sufficient only because the code paths THEY exercised never called another `Math` method. Adding a new `mutationFloor()` (which calls `Math.round`) to `loop-flutter.workflow.js` immediately exposed this gap in a fresh inline sandbox. **Apply:** any new inline VM sandbox for a Workflow script should build the full `Math` proxy from the start (`Object.getOwnPropertyNames(Math)`, override only `random`), not a single-method stub — copy the shared helper's shim rather than re-deriving a minimal one per test.
- **A multi-agent write→verify handoff should cross-check the write agent's own report of what it did, even when a downstream gate would catch the worst-case failure anyway.** The Minimize write-agent returns `{ removed: [...], originalContent }`; the orchestrator's real safety net is the confirm re-gate's exact-kill-count compare (which protects correctness regardless of exactly which/how-many tests were actually removed). But without cross-checking `removed.length` against the proposed count, a partial or off-target removal would silently corrupt the REPORTED `removed`/`restored` numbers even on a technically-safe run. Caught via a `boy-scout` pass, not the original design — a cheap `WARNING` log closes it. **Apply:** when one agent's structured return both drives a decision AND feeds a human-facing report, verify it against what was actually asked for, even if a later stage's gate would independently catch a truly dangerous failure — reporting accuracy and safety are separate properties.
