# F1-NotesPlugin — Lessons

## Architect Lessons

- **A "port from X" step needs a Keep/Change/Scrub list whenever X is bigger than the consumer's need.** Step 3 (search-notes) had one and survived review clean; Step 2 (note-schemas) said only "port from the live file" and drew the single HIGH — the 534-line producer contract would have shipped 26 dangling hub references. The under-specified port is the same defect class as the unscoped "replace all X."
- **Grep gates keyed to a repo-name literal miss hub-internal coupling.** The note-schemas source contains zero `omnishelf` literals yet 26 hub-only paths (`.claude/agents/…`, `kb/…`, `docs/proposals/…`) — G1 was structurally blind to all of them. When gating "no foreign coupling," grep for the *reference shapes* (path prefixes), not the repo name. G4 added.
- **Owner supersede of a ratified proposal at shaping:** record it in the spec's Source line and the extracted ADR's provenance blockquote — never edit the foreign repo's proposal unilaterally. The ADR-28 graduation chain stays intact with the supersede explicit at every hop.
- **External-repo grounding on this box:** Glob/Grep return empty for paths outside the session's working dirs (omnishelf-pipeline) while Read with absolute paths and PowerShell work — recurrence of the known sandbox-fabrication hazard; PowerShell remains the verification channel for external filesystem facts.
- **Concurrent-tree contamination reaches the twin:** gen-omni mirrors the working tree, so a dirty concurrent feature (adhoc-MineSkillAuthoring) contaminates a twin regen — the clean-tree guard became a plan-level gate (Step 6) instead of tribal knowledge.
- **Writing acceptance as executable grep gates + a pre-authored split-AC made the done-check fully deterministic — zero judgment calls.** Every step's load-bearing claim was verifiable by re-running the plan's own G1–G4 greps and the lint/unit suites (all green, 511 combined) against live files rather than trusting the self-report. The one deferral (AC6 live-twin half) resolved in a single line as `Deviated (valid reason)` precisely because the plan pre-authored the operator-owed fallback with an `OPERATOR ACTION REQUIRED` block and split AC6 satisfaction — the "pre-authored operator-owed fallback resolves in one line at done-check" and "acceptance asserts the mechanism" rules validated end-to-end on a shared-artifact pass.

## Reviewer Lessons

- **G1/G4-shaped grep gates catch path/hub-ref leakage but not value-level contract contradictions.** `note-schemas.md` defines `confidence` as a strict `integer (1–10)` with no missing-value fallback (unlike `completeness`, which states one explicitly); `search-notes/SKILL.md`'s own worked example and output-format template carried over a `pending` confidence value and a `Pending validation` count bucket from the omnishelf-docs source, verbatim, with no defined trigger anywhere in the shipped contract. Neither G1 (path literal) nor G4 (hub-ref shape) can catch this class of defect — it's a *value*-shaped leftover, not a path- or reference-shaped one. When a plan's port step says "cite the schema instead of restating it," the review should specifically diff the skill's worked examples/output templates against the schema's stated types and enums, not just grep for banned path/ref shapes. Worth a future grep-gate idea: flag skill-body example rows containing a token not in the cited schema's declared enum/type for a field (harder to automate cheaply, so likely stays a manual review check rather than a new G-gate).
- **A spec's own AC language can bake in an ambiguity the plan then inherits.** AC wave-1 #2 says search-notes "resolves **its** root" (singular) even though the cited config contract (`notes-config.md`) is explicitly a multi-source, per-source-`inbox`, `enabled`-flag list with "disabled sources are skipped by all skills" — implying iteration the skill body never states. Because the ambiguity is traceable to the spec's own phrasing (not just a developer omission), confidence on "is this a real gap" stays in the 60s — correctly routed to Open Questions rather than asserted as a Finding, per the ≥80 cutoff. Confirms the cutoff does its job: this is exactly the "developer/architect may hold context I don't" case the self-audit step is for.

## Developer Lessons

- **Phase-1 analyze re-grounding: the note-schemas producer source drifted within the plan-authoring day.** Plan (2026-07-12) characterizes `omnishelf-pipeline/conventions/note-schemas.md` as 534 lines with 26 hub-internal refs; live read same day is **468 lines / 20 hub-ref lines** (BOM + 5 mojibake em-dashes + 0 `omnishelf` literals all still hold). This is exactly the BR9 "producer surface keeps evolving" case — Step 2 must read the file live and scrub *whatever* hub refs exist, not a fixed 26. G4's "zero hits" gate is count-independent, so the drift is absorbed without a plan change. Confirms: never treat a port step's source-size numbers as a scrub checklist — they're a snapshot, the gate is the contract. (Implementation confirmed: Step 2 was authored as a **fresh** consumer contract, not a scrub-in-place port, so the source's BOM/mojibake/hub-refs never had a channel into the shipped file — the cleanest way to satisfy G3/G4 on a drifting source is to re-author, not to copy-then-strip.)
- **Shared references in a multi-skill plugin need the sibling-citation form, or skill-lint E6 fails.** When two skills in one plugin share references living in ONE skill's `references/` folder (the mine-family-core pattern, adopted here for `notes-config.md`/`note-schemas.md` in `search-notes/references/`), the *owner* skill cites `references/X` (resolves skill-relative) but a *sibling* skill must cite `../owner-skill/references/X`. A bare `references/X` from the sibling makes skill-lint E6 report a dangling reference (it resolves only skill-relative or at repo-root). The `../owner/references/X` form is also what skill-lint's E6 lookbehind `(?<![\w/])` exempts from the dangling check — so it is both the correct pointer and lint-clean. Precedent: `mine-verify-repo`/`mine-design`/`mine-algorithm` all cite `../mine-verify-cover/references/mine-family-core.md`.
- **When a plan maps `improve-skills` to a step and scopes the done-condition to skill-lint, follow the plan — do not also run the Judgment Gate.** improve-skills' New-Skill recipe makes the `evaluate-skill` Judgment Gate "mandatory" for a new shipped skill, but (a) the plan is authoritative on feature-specific decisions and Step 3/4 accept explicitly = "skill-lint green" + grep gates, and (b) `evaluate-skill` would be a pipeline-agent spawn the developer role must not initiate. skill-lint exit 0 is the honored done-condition; note the scoping as a Key Decision.
- **`bump-plugin.mjs` already handles the new-plugin case — no manual revert needed.** release-plugin's New-Plugin checklist item 1 warns you must revert a `0.1.0 → 0.1.1` re-bump on a brand-new plugin. The current tool doesn't propose that: `--dry-run` prints `new plugin (absent at HEAD) — first release ships at its authored version; no bump` and applies nothing. So on a first release the developer runs `--dry-run` to confirm the classification is clean/F1-scoped and there is nothing to apply — the checklist warning is now defused by the tool, not by hand.
- **Fix-cycle decision (Step-2 review MEDIUM, `search-notes` "pending" confidence):** when a review finding poses "add a fallback vs. strip the stale case," ground the decision in the **producer's own agent source**, not the doc being ported from — the ported doc can itself be stale relative to its own producer. Here `note-schemas.md` (both the omnishelf-docs source skill's era and the ported reference) never documented a confidence fallback, but the deciding evidence was `notes-extractor.md:223,273` (the live producer agent card) explicitly banning the string label `pending` — proof the pipeline never emits an unvalidated confidence, so a fallback would document an impossible case. Stripped the stale bucket/example instead of inventing a fallback; swept the whole plugin (`grep -rn "pending"`) to confirm the defect was isolated, not half-landed elsewhere (AP2).
