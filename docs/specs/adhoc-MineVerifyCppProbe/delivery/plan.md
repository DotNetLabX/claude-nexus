# Mine→Verify→Cover — C++ adapter Phase-0 probe

**Feature Spec:** None (ad-hoc technical de-risk probe)
**Parent:** `docs/specs/adhoc-MineVerifyCoverHarness/roadmap.md` — "Multi-language end goal" (C++ adapter, deferred).
**Governing decisions:** ADR-25 master gate (prove the loop before paying the build); the harness's language-neutral-core / per-stack-adapter split.
**Research backing:** `docs/kb/research/cpp-mutation-and-test-tooling.md` (mull = Stryker.NET equivalent; libclang; GoogleTest+RapidCheck).

## Context

The harness ships a generic core skill (`nexus/skills/mine-verify-cover`) + per-stack adapter (`nexus-dotnet/skills/mine-verify-cover-dotnet`, the only adapter so far). The owner wants to know whether it can port to the embedded-C++ SDK at `D:\omnishelf\omnivision-ai-sdk`. Research resolved the load-bearing unknown positively: a real C++ toolchain exists (mull / libclang / GoogleTest). What remains unproven is whether the harness's 5 adapter capabilities actually run end-to-end on a real slice of that SDK — the substrate has no test framework, no CTest, a Linux-only host build, and unstable static teardown.

This probe de-risks the **C++ adapter** before committing to build it. It is a two-way-door spike, **not** a decision to sequence C++ ahead of the roadmap's Flutter increment — that ordering stays the owner's call.

## Scope

**In scope:** prove all 5 adapter capabilities + one full Mine→Verify→Cover pass on **one** dependency-light slice, under **WSL2 Ubuntu**, producing a GO/NO-GO verdict with cost numbers.

**The slice (pinned):** `D:\omnishelf\omnivision-ai-sdk\src\trackers\sortcpp\hungarian.{h,cpp}` — 419 LOC, pure C (`<stdio.h>`/`<stdlib.h>` only), high branch density (`hungarian_solve`: nested loops, goto-augmentation, slack/threshold conditions). Public API: `hungarian_init` / `hungarian_solve` / `hungarian_free`. Fallback slice: `src/3rdparty/dbscan/dbscan.{h,cpp}` (zero-dep, smaller surface) if Hungarian blocks.

**Out of scope:** full adapter extraction/authoring; multi-class sweep; building the whole SDK (heavy native deps + hardcoded dev paths stay untouched — we isolate the 2 slice files into a throwaway CMake project); Windows-native (mull is Linux/macOS only); modifying the SDK repo (slice files are **copied** into the probe workspace, SDK stays pristine).

## Implementation Steps

### Step 1 — WSL host + isolated slice build under Clang
Stand up WSL2 Ubuntu; install Clang/LLVM (a major mull ships a package for — e.g. LLVM 18). Create a **throwaway** CMake project (NOT inside the SDK; e.g. `~/cpp-mvc-probe/`) that compiles the **copied** `hungarian.{h,cpp}` as a standalone library target, `CMAKE_CXX_COMPILER=clang++` (the file is C — compile as C or C++), `-DCMAKE_EXPORT_COMPILE_COMMANDS=ON`.
- Files: `~/cpp-mvc-probe/CMakeLists.txt`, copied `hungarian.{h,cpp}`.
- **Accept:** library builds clean under Clang; `compile_commands.json` emitted; grep of the compile DB shows **zero** `opencv`/`ncnn`/`Eigen`/`openexr`/`curl` includes (proves isolability). Mechanism: `cmake --build` exit 0 + `grep -iE 'opencv|ncnn|eigen' compile_commands.json` returns nothing.
- **Satisfies:** probe exit-criterion 1 (host + slice compiles). Confidence: medium (LLVM/mull version-pin friction is the known WSL hazard).

### Step 2 — GoogleTest + CTest, proven green-twice
Add GoogleTest via CMake `FetchContent`; write ONE seed test calling `hungarian_init/solve/free` on a small cost matrix with a hand-verified optimal assignment; register with `gtest_discover_tests`; run `ctest` **twice**.
- Files: `~/cpp-mvc-probe/CMakeLists.txt` (FetchContent block), `~/cpp-mvc-probe/tests/hungarian_smoke_test.cpp`.
- Pattern reference: research entry `## Finding` (GoogleTest + `gtest_discover_tests`).
- **Accept:** `ctest` green on both runs with identical pass counts — proves the harness's `suite_green` + `no_flaky` gates have a real runner. Mechanism: two `ctest --output-on-failure` runs, both exit 0, same test count.
- **Satisfies:** exit-criterion 2.

### Step 3 — mull mutation gate, proven kill + parseable report
Install the LLVM-pinned mull package matching Step 1's Clang. Add `mull.yml` (`compilationDatabasePath` → the Step-1 `compile_commands.json`; `reporters: [Elements, SQLite]`; `workers`). Recompile the slice + test with `-fpass-plugin=mull-ir-frontend`; run `mull-runner` headless against the gtest binary.
- Files: `~/cpp-mvc-probe/mull.yml`, build-flag wiring in `CMakeLists.txt`.
- **Accept:** mull runs non-interactively, kills **≥1** mutant on `hungarian_solve`, and emits a parseable `mutation-testing-elements` JSON **and** SQLite DB from which a kill-score float is extractable. Mechanism: a 5-line parser reads the JSON `killed/total` and prints a ratio.
- **Satisfies:** exit-criterion 3. Confidence: **low-medium** — the `-fpass-plugin` + exact LLVM-major match is the highest-friction point in the whole probe; if mull won't attach, fall to the research entry's dextool fallback note (heavier) before declaring NO-GO.

### Step 4 — libclang evidence slice (Mine input)
`pip install libclang`; write a small `clang.cindex` script that reads `compile_commands.json` and extracts the source extent (signature + body) of `hungarian_solve` (and the other public fns), filtered to the main file.
- Files: `~/cpp-mvc-probe/index_slice.py`, output `~/cpp-mvc-probe/evidence/hungarian.slice.md`.
- Pattern reference: research entry `## Fix` (libclang cursors + `clang_Location_isFromMainFile`).
- **Accept:** a per-function evidence slice is produced (the Roslyn-equivalent Mine input). Mechanism: the slice file contains the `hungarian_solve` body text and no transitive-header noise.
- **Satisfies:** exit-criterion 4.

### Step 5 — One end-to-end Mine→Verify→Cover pass on the slice
Instantiate the harness loop (the `mine-verify-cover` method, run via the existing Workflow scripts `harness/mine-verify.workflow.js`, `harness/cover.workflow.js`, `harness/loop.workflow.js`) with a **C++ adapter shim** wiring the three toolchain seams proven in Steps 2–4: evidence indexer = Step-4 slice, test runner = ctest/gtest, mutation tool = mull. Target: `hungarian_solve`.
- Files: a thin adapter config/shim (location: `docs/specs/adhoc-MineVerifyCppProbe/delivery/` for the config record; the live wiring runs in the WSL workspace). Harness-generated tests land in `~/cpp-mvc-probe/tests/`.
- **This is the real de-risk** (no precedent — the harness has never run on C++): does clean-room Mine produce rules from C source, does Verify refute, does Cover **generate gtest tests that compile + run + kill mull mutants**, do the gates compute.
- **Accept:** ≥1 verified rule; a mull kill score on harness-**generated** tests; the gate battery computes at least `suite_green` / `no_flaky` / `mutation_floor`; **cost recorded** (output tokens + wall-time). Mechanism: the loop's own report artifact + the captured token/time numbers.
- **Satisfies:** exit-criterion 5. Confidence: **low** (the whole point of the probe).

### Step 6 — GO/NO-GO verdict + cost + adapter findings
Write the probe report: per-capability result (Steps 1–5), the kill score, cost vs the .NET per-class ceiling (~200–260k output tokens, ~30 min), the friction points (Clang requirement, LLVM/mull pinning, Windows→WSL, unstable static teardown if it surfaced), and a **GO/NO-GO** recommendation for building the full C++ adapter. Record the standing finding that **domain-rich slices (planogram KPIs, fixing_tools) are OpenCV/Eigen-trapped** → the real adapter must budget for slice-isolation, not just toolchain wiring.
- Files: `docs/specs/adhoc-MineVerifyCppProbe/delivery/implementation.md` + `docs/specs/adhoc-MineVerifyCppProbe/delivery/probe-report.md`.
- **Accept:** a binary GO/NO-GO with evidence per capability and the cost numbers.
- **Satisfies:** exit-criterion 5 + the probe's decision output.

## Cross-Service / Cross-Host Notes

- **Cross-host:** all build/run steps execute in **WSL2 Ubuntu** (mull constraint). The Windows side holds only the SDK source and the captured artifacts.
- **Cross-repo:** the SDK (`D:\omnishelf\omnivision-ai-sdk`) is **read-only** — slice files are copied into the throwaway WSL workspace; the SDK repo is never modified. Probe artifacts + the report are captured back into this nexus delivery folder.
- **No SDK CI/CTest exists** — the probe stands up its own isolated CMake/CTest; it does not wire anything into the SDK.

## Testing Strategy

The probe's "tests" are the **gate proofs themselves**: Step 2 proves the test runner (green-twice), Step 3 proves the mutation gate (≥1 kill + parseable report), Step 5 proves the harness-generated tests compile/run/kill. There is no separate hand-written test suite — that is the harness's job and is exactly what Step 5 de-risks.
