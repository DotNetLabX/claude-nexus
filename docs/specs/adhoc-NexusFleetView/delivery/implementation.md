## Key Decisions
- **Step 1.0 root field — `data.workspace.project_dir` (confirmed via official statusLine docs, not a
  live spike).** A `subagentStatusLine` runs in the main session, not a subagent sandbox, so a
  subagent structurally cannot capture its payload — the team lead owns the post-build live confirm.
  Resolution per handoff: the documented top-level statusLine payload field `workspace.project_dir`
  is the launch root. Deliberately **not** `process.env.CLAUDE_PROJECT_DIR`/`CLAUDE_PLUGIN_ROOT`
  (hooks-only env, absent for a statusLine process), **not** `tasks[].cwd` (a per-task subagent dir →
  wrong audit location), and **not** bare `process.cwd()` (ADR-8 stray-log). No root → the write is
  skipped (fail-open; row rendering unaffected). `tokenCount`/`startTime` presence was already
  established by the shipped row renderer. This deviates from the plan's literal Step 1.1 prose
  (`root = process.env.CLAUDE_PROJECT_DIR || <field>`) — see Deviations.
- **Time seam for deterministic tests.** `render-fleet.mjs` computes elapsed/staleness against an
  injectable `now` (env `NEXUS_FLEET_NOW`, default `Date.now()`), so the verbatim-string fixtures are
  stable. The heartbeat itself needs no seam (it stamps `new Date().toISOString()` and tests read the
  written shape, not a time-derived string).
- **Join key = role, normalized.** The token-usage `agent` field is messy (`nexus:developer`,
  `developer-2`, `developer`, `main`, …). `roleOf()` maps each through the same token-boundary role
  match the row renderer uses, so all developer aliases fold to `developer` and non-roles (`main`)
  are excluded — matching the row renderer's "only nexus roles" behavior.
- **Dedup replicated, not imported (plan M2).** The consumption-report consecutive-tuple dedup (same
  key `[input,output,cache_read,cache_creation]`) is copied into `readDepth`, since the original lives
  inline in a `node -e` block. The fleet line surfaces only `calls`, but `out` is computed and
  asserted equal to the consumption-report result, pinning parity.

## Deviations from Plan
- **Step 1.1 root resolution dropped the `CLAUDE_PROJECT_DIR` fallback.** The plan's literal prose
  was `root = process.env.CLAUDE_PROJECT_DIR || <confirmed field>`. The handoff (grounded in the
  official statusLine docs, comm-log message #4) established that `CLAUDE_PROJECT_DIR` is **not** in
  the statusLine env — it is hooks-only — so in production it is always undefined and the fallback is
  dead. Implemented root resolution from `data.workspace.project_dir` only. This is the plan's intent
  (a usable project root, never a subagent dir or stray cwd), corrected for the confirmed contract.
  Tests inject the root via the payload's `workspace.project_dir`, not the harness `projectDir` env,
  to mirror the real statusLine.
- **SKILL.md does not write the `/nexus:fleet` command token.** The plan/handoff refer to the trigger
  as `/nexus:fleet`. The repo's wiring lint reserves `nexus:<name>` tokens for agent-backed commands
  (resolves them to `agents/<name>.md`), and no shipped skill writes its own `/nexus:` token in its
  body (e.g. `consumption-report` doesn't). To stay lint-clean and convention-consistent, the SKILL.md
  refers to it as "the fleet view"/"the `fleet` skill"; the user still invokes it as `/nexus:fleet` at
  the prompt (the standard skill-trigger mechanism). No behavior change.
- **`description` frontmatter rephrased to avoid a YAML colon-space.** `claude plugin validate
  --strict` parses frontmatter as real YAML; an unquoted plain scalar cannot contain `": "`. The
  original description had `pipeline agents: a run-state header` — rephrased with em-dashes/commas so
  it parses. (The repo's lenient flat-regex `frontmatter()` accepted it; the strict validator did not.)

## Cycle-2 LOW Cleanup (2026-06-16)

Both reviews (nexus reviewer APPROVE, Codex GO) returned no blockers; this is the consolidated LOW
fix-list from the team lead. Full suite stayed green (193/193, +1 for the new zero-token test);
skill-lint clean; `validate --strict` clean; omni twin regenerated and in sync.

- **LOW-1 — fmtK zero-case divergence (fixed).** `render-fleet.mjs` `fmtK` returned `'0'` and the
  render site forced `'0 tok'`; now `fmtK` returns `''` for zero/absent and the caller omits the
  token field entirely, matching `subagent-rows.js` so a row never shows `0 tok`. Pinned by a new
  `zero-token` fixture + verbatim test (t1 `tokenCount:0`, t2 absent → neither renders a token field).
- **LOW-2 — dead Windows main-guard (fixed).** Replaced the `import.meta.url === \`file://${argv[1]}\``
  comparison (dead on Windows; the `endsWith` fallback was doing the work) with
  `import.meta.url === pathToFileURL(process.argv[1]).href`, which handles the drive-letter/separator
  quirks. Removed the misleading fallback. CLI invocation and the import-without-running-main path both
  verified.
- **LOW-3 — null-role rows undocumented (fixed).** Added a code comment at the per-agent render that
  null-role tasks render as `[?]` on purpose — the fleet view shows the whole roster (unlike the row
  renderer, which hides non-pipeline rows). Full visibility is intent.
- **LOW-4 (Codex) — dedup consecutive under-count (NOT fixed by direction; documented).** Kept the
  algorithm identical to `consumption-report` per plan M2; added an in-code comment stating the known
  limitation (a role's dedup key persists across other roles' interleaved rows, so two distinct
  same-tuple turns of one role separated by another role can collapse). The fix is the deferred
  shared-module extraction, not a divergence here.
- **Misleading test comment (fixed).** `subagent-rows.test.mjs` malformed-stdin test: clarified that
  `projectDir` sets `CLAUDE_PROJECT_DIR`, which the script deliberately ignores — the test passes
  because the env root is not a root source AND the payload never parses.
