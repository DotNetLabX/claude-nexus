# Cross-Check Review — adhoc-BuildFlowFormalization

**Verdict:** NO-GO

## Summary
The change violates its own “reference P1/P2/P3; never restate them” constraint in multiple normative files, and the new proposal/backlog flow has unresolved schema contradictions. ADR numbering, cited line references, and the new `Satisfies:` wording itself checked out; the blocking issues are in the content and cross-references.

## Findings

| # | Severity | File | Issue |
|---|----------|------|-------|
| 1 | BLOCKER | `docs/specs/adhoc-BuildFlowFormalization/delivery/plan.md`; `docs/architecture/README.md`; `docs/specs/adhoc-BuildFlowFormalization/definition/tech-spec.md`; `plugins/nexus/skills/proposal-format/SKILL.md` | The pass declares a hard constraint to “**Reference P1/P2/P3; never restate them**” ([plan.md:18-21]) and ADR-26 repeats that P1/P2 “**are referenced by name; their entry/output schemas are not restated here**” ([README.md:690]). But the change then re-describes those schemas anyway: “recommendation + confidence + options eliminated” ([plan.md:113-115], [README.md:690], [tech-spec.md:96-97]) and proposal-layer additions/reused fields such as `impact`, `effort`, `recommendation`, `confidence`, and `alternatives` ([tech-spec.md:129-132], [proposal-format/SKILL.md:14-16], [proposal-format/SKILL.md:35-39], [proposal-format/SKILL.md:56]). This is an explicit violation of the stated hard constraint. |
| 2 | MAJOR | `docs/specs/adhoc-BuildFlowFormalization/definition/tech-spec.md` | Step 2 says the tech-spec must keep “the rationale home” and “**Do not duplicate the ADR decision text — link to it**” ([plan.md:158-167]). The current tech-spec still re-states ADR decisions instead of only linking: the mandatory/optional stage rule ([tech-spec.md:86-91]), the RESEARCH stage behavior/output ([tech-spec.md:93-97]), the architect-vs-PO definition split ([tech-spec.md:105-109]), and the proposal-layer additions / research-first rule ([tech-spec.md:125-132]). That conflicts with both the plan acceptance and ADR-27’s “one authoritative source” rule ([README.md:714-718]). |
| 3 | MAJOR | `plugins/nexus/skills/proposal-format/SKILL.md` | ADR-28 defines proposal front-matter with `Impact: 1–10` ([README.md:733]) and ADR-29/backlog ranking assumes each ratified proposal has impact and effort for ordering ([README.md:754], [docs/backlog.md:5-6], [docs/backlog.md:15]). The skill weakens that to “`Impact:` … **omit for pure-internal plumbing**” ([proposal-format/SKILL.md:27]). That creates a format variant the ADRs/backlog model do not support and leaves ratified internal proposals without the field the backlog ranking requires. |

But the new backlog schema has only `Slug / Title`, `Source proposal`, `Impact`, `Effort`, `Impact ÷ Effort`, and `Status` ([docs/backlog.md:15-16]; planned schema at [plan.md:200-202]). There is no column or documented field for the required spec link. |
| 5 | MAJOR | `docs/backlog.md` | The illustrative row points at `docs/proposals/sample-feature.md` ([docs/backlog.md:17]), but that file does not exist on disk. Under the review contract, a referenced missing file is a broken cross-reference. |
