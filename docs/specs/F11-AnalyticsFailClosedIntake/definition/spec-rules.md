## Spec findings

1. **SF-1 — "One declaration shape" (SR-12/BR12) has no stated shape for one of its two halves.** The spec itemizes concrete fields for question-level inputs (Identity, Ask guidance, Resolution policy — one of three named enum values) but gives per-measure flags no analogous field breakdown anywhere — only their aggregation behavior (additive/union) is stated. BR12's "the shape defined by this spec is the contract" is therefore verifiable for the question-level half and unverifiable for the per-measure half; the spec also never disambiguates whether "shape" means this conceptual field set or a concrete serialization (file format, key names) — no such serialization is committed anywhere. **Disposition: open** — recommend the spec either point to a fuller per-measure-flag field list or explicitly defer exact serialization to plan-time (as BR12's SDD-lifecycle sibling artifacts do for other "shape" questions), the way F8's SR-30 folded a parse-floor onto "machine-consumable."

2. **SF-2 — The grain-ladder failure signal is asserted, never described (SR-34).** BR5/Entities/AC-6 all commit to a response when "the ladder cannot establish a grain," but the spec never states the mechanism or criterion by which the ladder itself determines or communicates that failure — it defers entirely to "the model's existing resolution ladder," a mechanism defined outside this document. All 3 independent miners flagged this same gap unprompted. **Disposition: open** — this is a real spec gap, not a defect in the mining; the response is testable as a black box (given an unresolvable-grain scenario, expect a business-language ask), but no spec text lets an implementer or tester construct that scenario from first principles, or verify the ladder's own decision boundary.

3. **SF-3 — The legacy-file discovery mechanism is asserted, never described, and sits in tension with BR6 (SR-35).** BR6 commits the plugin to shipping "no product-specific input (no 'chain'")," yet Flow 5 and the Entities legacy-migration paragraph require reading a chain-specific value from "an untracked repo-local config file" — without ever stating how an input-agnostic plugin locates, names, or parses that product-specific file. All 3 independent miners flagged this same gap unprompted, and 2 of 3 explicitly noted the tension with BR6/BR9's agnosticism claims. **Disposition: open** — either the declaration data itself must carry a per-product legacy-source pointer (making discovery agnostic-by-declaration, consistent with BR6's "everything else comes from declaration data"), or the spec should say so explicitly; as written, the mechanism is a gap the spec assumes away.
