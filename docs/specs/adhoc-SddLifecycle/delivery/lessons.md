# SDD Golden-Set Lifecycle — Lessons

## Architect Lessons

- **A technical feature gated on a *sibling* pilot's live verdict decomposes along the gate — surface the gate
  as the #1 owner question, never assume past it.** This spec was `Status: Ready` but explicitly gated
  implementation on the parent pilot's AC-6, which was pending (verified: parent `summary.md`, no `.runs/`
  evidence). The right Phase-1 move was to split the feature into an **ungated slice** (safe regardless of the
  verdict — here M2/M0 + drift rules) and a **gated slice** (invalidated by a NO-GO — C1–C4/M1/M3), and put
  the "how much to build now" choice to the owner with a master-gate (ADR-25) read. Reusable pattern for any
  "implementation gated on X's verdict" spec: enumerate what's true-regardless vs what-X-could-invalidate, and
  let the owner pick the scope — don't silently plan the full build.
- **The architecture README `## Contents` list is not authoritative for the next ADR number — grep `^## ADR-`
  for the real highest.** The Contents TOC ended at ADR-36, but a live `grep '^## ADR-3[0-9]'` found **ADR-37**
  already extracted (MVC Minimize). Nearly assigned 37/38 to new ADRs; the grep corrected it to 38/39. Always
  resolve a "next sequential ID" from the live section headings, not the table of contents (it lags).
- **A plugin-editing pass hands the developer a `selfcheck` gate that cannot pass — `gen-omni --check` is
  gating, and the twin regen is team-lead-owned at commit time.** `scripts/selfcheck.mjs:73-77` runs
  `gen-omni --check` as a **gating** check (not `informational`), and it exits 1 on any twin drift. But the omni
  twin is regenerated by the team-lead **after** the last nexus edit (ADR-6 + memory "Finalize artifacts before
  commit 2"), so at the developer's stage the twin is stale by design → `selfcheck` can never be fully green for
  a `plugins/**`-touching feature. A plan that says "selfcheck exits green" as developer acceptance is
  unsatisfiable and risks the same bounce class as the documented `gen-commands` false-positive
  (`selfcheck.mjs:54-63`). **Rule:** when a plan touches `plugins/**`, scope the developer-stage release
  acceptance to `validate --strict` + the *non-omni* selfcheck gating checks, and mark `gen-omni --check` as an
  **expected-RED known-deferred** state owned by the team-lead — never a developer-green gate. (First hit here;
  the parent `SddCoverageLoop` never saw it because it changed no `plugins/**` files.)
  **Evidence:** [adhoc-SddLifecycle]. **Priority:** medium — recurs on every future plugin-editing plan.
- **In this repo the shipped harness is dev-repo-only — confirm what actually ships before scoping a "fold-in".**
  `harness/**` lives at the repo root (not under `plugins/`), so the SDD-loop `.mjs` modules never reach users;
  the fold-in's shippable surface is **skill text + agent files** only. This flips the plan from "build modules"
  to "author skill/agent prose + release" and makes this the first `plugins/**` touch (bump + omni twin) in a
  lineage whose parent shipped nothing. Check the ship boundary (repo-root vs `plugins/`) before assuming a
  harness feature ships code.
