## Developer Lessons
- **`bump-plugin.mjs --check` flips to "no bump needed" *after* you apply the bump — that is the green state, not a regression.** The check is git-HEAD-based: post-`bump-plugin.mjs`, the working-tree `plugin.json` already differs from HEAD, so the check reads "the bump is present" → exit 0. Don't re-bump or panic when `--check` says "no bump needed" right after applying one; it means the bump is staged-ready.
- **`claude plugin validate --strict` currently fails on two *unrelated, pre-existing* skills** (`evaluate-skill`, `improve-skills`) over their long multi-line `description:` frontmatter + `user-invocable` key. Before attributing any validate failure to your change, run `git diff --name-only HEAD | grep <file>` — if the failing file isn't in your diff, it's pre-existing. The authoritative project gate is `scripts/selfcheck.mjs` (lint+unit), not `claude plugin validate`; selfcheck's lint+unit passed clean.
- **The two `selfcheck` FAILs at a developer's pre-commit stop (`gen-commands drift`, `gen-omni --check`) are both expected and both owned downstream of the developer.** gen-commands drift is the documented HEAD false-positive (commands match working-tree agents; resolves at the team-lead commit). gen-omni drift is owner follow-through (regen + `../omni` commit happen after the bump, by the owner). A developer should regenerate commands (done), confirm the command diff matches the agent diff line-for-line, and NOT run gen-omni or commit.
- **For an ADR-30 additive-discipline prose pass, "verify the byte-unchanged claim" is a grep of the Pre-Flight numbering, not a re-read.** After collapsing #1+#2, `grep -nE '^[0-9]+\.|^4b\.'` instantly confirmed the sequence stayed `0,1,3,4,4b,5,6,7` and that the `4b` cross-reference target survived — cheaper and more reliable than eyeballing the whole block.

## Architect Lessons
- **A guard/hook behavior claim in a plan overlay must be grounded against the regex, not the doc-comment.** Step 1 originally justified the stale-default fetch overlay with "hardened mode blocks remote fetches (README / guard.js)" — both the README:40 table and the guard.js:10-12 doc-comment *say* "fetches," but the actual hardened block set (`guard.js:138-144`) is `git push` + network installs + `curl|wget|iwr|invoke-webrequest`; **no regex matches `git fetch`.** The developer caught it; the plan was wrong because it trusted the prose self-description of the hook instead of the matcher. When a plan leans on a hook's enforcement, read the *regex/predicate*, not its header comment — and the safe reframe for any best-effort step is "fails for any reason → warn-and-skip," which subsumes the policy case without asserting a specific (and here false) enforcement mechanism.
- **"Defer to hardened guard mode" is only actionable if the agent can detect the mode — and it can't.** `security_mode` is install-time userConfig injected as `guard.js` argv[2] via `hooks.json:15`; there is no env var or file an agent can grep at runtime. An instruction that tells an agent to behave differently *based on* the guard mode is unactionable by construction. The push gate (Step 3) survives because it doesn't ask the agent to detect the mode — it just lets the guard reject the push and surfaces that; the fetch overlay's original wording wrongly asked the agent to pre-defer. Prefer "let the enforcement layer reject and handle the failure gracefully" over "agent, check the policy first."
- **Done-check token scoping: `.claude/.pipeline-state` holds the *current* phase token (here `architect:donecheck`), not the developer's.** To score skill conformance, filter the log by `agent:developer` + `token:developer:implement` and match on the final path segment (`nexus:release-plugin` → `release-plugin`) and the run's date/session — not by the live pipeline-state token, which is mine, not the developer's.

## Reviewer Lessons
- **For a prose/agent-doc pass, `git diff HEAD` is the primary review instrument, not file reads.** The diff directly confirms ADR-30 byte-unchanged discipline (no lines removed that weren't sanctioned), numbering stability, and additive placement — facts that are hard to verify from reading the full file but immediate from the diff's `+`/`-` lines.
- **Carry-over verification is a boolean check against `git diff --name-only`, not a re-read of the failing file.** A pre-existing selfcheck FAIL that only affects files absent from `git diff --name-only HEAD` is confirmed pre-existing in one command — no need to open the file.
- **For "canonical rule, no matrix duplicated" acceptance criteria, a grep for matrix keywords (e.g. "Attended | Unattended") in the referencing file is faster than reading the whole file.** Solo's lightweight character was confirmed in one `grep -n` call that showed no table rows.

## Team Lead Lessons
- **Finalize every slug artifact (communication-log close-notes, lessons.md) BEFORE making the implementation commit — not after.** This run took a 3rd commit (against the 2-commit default) because the comm-log close-notes were written *after* commit 2, and the obvious fix — amend commit 2 — was blocked: the omni twin's footer pins the nexus impl sha (`Generated from the nexus plugin (nexus <sha>)`), so amending commit 2 would orphan that reference. **Why it matters:** because the omni footer pins the impl sha, commit 2 must be the *last* edit of the nexus tree before the omni sync — any later artifact edit forces BOTH a commit-2 amend AND an omni footer re-point. **How to apply:** sequence the close as (1) write summary.md + comm-log final state + lessons.md, (2) commit 2, (3) `gen-omni` + omni commit referencing commit 2's now-final sha. **Recovery if a stray post-commit-2 edit happens anyway:** `git reset --soft <impl-sha>` then `git commit --amend --no-edit` to fold the edit back into commit 2, then `git commit --amend` in `../omni` to re-point its footer to the new sha — restores the 2-commit shape without losing provenance. (This very lesson was folded into commit 2 via that recovery, not added as a 4th commit.)
