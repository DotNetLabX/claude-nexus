## Architect Lessons

- **Same-session oracle production is a structural fault, not a sequencing accident.** Pilot 2 followed the then-ratified Graduate-to-spec text ("its pilot pulls a Mine→Verify registry first") and mined the clusterize registry inside the consuming run — colliding with a concurrent canonical registry run (truth fork, contained at SDK `128aab9` + supersede + brief renumber) AND compromising the conformance gate's independence even absent the collision. The SDK architect's proposal amendment (HARD BLOCK: stage-0 precondition, never self-mine) is the correct supersession; folded into plan step 2 the same day. Lesson: when a method consumes a verified artifact as its oracle, the plan must require the oracle to be produced by a *separate* run — "the pilot pulls it first" hid a same-session shortcut.
- **A ratified proposal can be amended mid-planning by the consuming repo's architect; re-ground the affected plan step immediately** (revision-pass rule) rather than letting the critic catch the drift — the binding contract is the live proposal text, not the version read at plan start.
- **A member/family-count edit is a counted-surface change — grep the whole shipping unit for external echoes of that count.** The plan grew the family table 5→7 and updated the counts *inside* the core reference, but three same-release sibling SKILL.md files echoed "all five"/"4-row" pointers (one already stale before this feature) that the step's file-scoped accept greps never covered. The developer caught it as a Phase-1 question. Lesson: when a step changes a count that other shipped files restate, enumerate those files from a grep at plan time and fold the sync into the same step — the accept grep must be scoped to the *named* files (a `mine-*` glob false-positives on new siblings that legitimately reuse the number on a different basis, e.g. a five-verdict grammar), and a cross-plugin echo becomes a recorded `deferred` decision, not silence.
- **"Follow improve-skills" for a NEW shipped skill silently pulls in the mandatory Judgment Gate — scope it explicitly in the step, don't leave the developer to discover the two-part done-condition.** Step 6 read as lint-only ("Deterministic Gate"), but the dev-repo carve-out binds a new shipped skill to lint + evaluate-skill (owner decision 2026-06-20; SKILL.md:21/:74-90). The Testing-Strategy "pilot-validated method" sentence covered the *method* axis and masked the missing *authored-artifact* axis. Lesson: whenever a plan step Follows a skill whose done-condition is conditional on the artifact being new-vs-existing, state which branch applies and pin its full acceptance — an under-specified "Follow X" hides the branch the developer must run.
- **Mechanism-named acceptance made the done-check fully deterministic — every step resolved by re-executing the plan's own greps, zero read-and-judge calls.** All 8 steps verified green in two batched commands (greps + lint re-run) because each accept line named its exact grep/command and expected count at plan time, including the negative pairs (row-9 `adapt, don't adopt` = 0; sibling `all five|[45]-row` = no hits) and the pre-scoped carve-outs (core `:44` dimension-count homonym, `mine-*`-glob false-positive note). Cost of the whole done-check: 2 file reads + 4 shell batches. Keep writing accepts this way — the one accept that needed a dispatch-note override (step 8's `git log -1` under a team-lead-owned commit) was exactly the one whose mechanism lived outside the developer's authority; pin the ownership split into the accept line itself next time ("bump greps developer-verifiable; commit accept deferred to close").
- **A partial-ownership step wants a split accept line.** Step 8 bundled developer-verifiable accepts (version, CHANGELOG) with a team-lead-owned accept (`git log -1` subject) under one Accept sentence; the dispatch had to carve it out in prose. When a step's Owner line says "team-lead/operator at close," split the acceptance into "verifiable now" vs "verifiable at close" so the done-check needs no external instruction to score it.

- **A proposal's "RESOLVED — the sibling/core already owns X" note is a dependency CLAIM, not a verified fact — grep the cited owner before a plan inherits it.** mine-design proposal line 104 resolved its census-at-scale Unresolved with "the mine-family core owns the 64k chunked-writes discipline (shipped nexus 1.26.1)"; the core never actually carried it (`grep chunk|64k` = 0). Plan step 4 inherited the claim ("chunked-writes discipline via family-core pointer") and shipped a dead pointer the reviewer caught (design-origin MEDIUM). Lesson: when a proposal marks an Unresolved item RESOLVED by pointing at an existing artifact, the plan step that depends on it must verify the target carries the content — a resolved-Unresolved note is exactly where an aspirational "someone owns this" hides. (This is the code-grounded-review-for-shared-artifacts mandate applied at *plan* time: the anchor was a shipped sibling file, so it had to be grepped, not trusted.)
- **A required-input precondition needs its absence ACTION stated, and the action's strength must match WHY the input is required — don't copy a sibling's gate by analogy.** mine-design named a Stage-0 "the BR registry exists" precondition but no absence behavior; the tempting fix was to mirror mine-algorithm's HARD-BLOCK never-self-mine. Wrong by one axis: mine-algorithm's registry is the adversarial *conformance oracle* (self-mine → truth-fork → gate independence destroyed), mine-design's is a census *input* (deletion oracle), not the judge's oracle — so the correct action is halt-and-route-to-produce (lighter), not never-self-mine (heavier). Lesson: two skills can share an input type and need *different* absence gates; derive the gate from the input's *role in the pipeline* (oracle-of-the-gate vs input-to-a-stage), never from "the sibling does X."

## Developer Lessons

- **Family-member counts drift across sibling SKILL.md pointers, not just the core doc.** A plan that grows the mine-family table must sweep every "all N members" / "N-row family table" pointer in sibling skills, not only the counts inside `mine-family-core.md`. Found a pre-existing stale "4-row" in `mine-verify-cover/SKILL.md` while analyzing this — the counts are demonstrably informal (a stale one already ships), which is itself the signal that a growth step needs an explicit scope decision on whether to sync them (surfaced as Q1). Generalizes the registry-guardrail "sweep every normative surface" (improve-skills AP2) to member-count facts. **Cycle-2 recurrence (Codex found it, the accept grep didn't):** a sibling can carry the family membership in TWO places — the top count-sync line (`N-row`/`all N`, which the Q1 accept grep validates) AND a second "Relationship to other skills" pointer that *enumerates a member subset* in prose. The count-based accept grep (`7-row|all seven`) is structurally blind to a stale member *list* — it checks the number, never the names — so both `mine-reference-model` and `mine-verify-repo` shipped a stale 3-member enumeration past a green grep. Lesson: when a growth step syncs member facts, sweep enumerated member *lists* too, not just counts; and prefer replacing a fragile prose subset with a count-word pointer (`all seven members`) or a bare "see the core §The mine family" so there is no subset left to go stale.
- **Glob does not reach an external repo that isn't a registered working directory; PowerShell is authoritative.** `Glob` with an absolute `path` into `D:\omnishelf\omnivision-ai-sdk` (not a working dir) returned "No files found" for files that exist. Confirmed present via `Test-Path`/`Get-ChildItem`. Companion to the existing memory "Bash sandbox fabricates external FS" — for external-repo FS facts, verify with PowerShell, don't trust an empty Glob/Bash result. **Phase-2 recurrence:** the pilot run-reports (which DO exist) came back empty from `Glob(path=SDK)` and were confirmed via PowerShell `Get-ChildItem` — same failure, second sighting.
- **Directory-segment brace globs are unreliable in the Grep tool; grep each file directly for a scoped-file accept check.** A `{mine-verify-cover,mine-reference-model,mine-verify-repo}/SKILL.md` brace glob returned "No matches found" for content that was present (verified by three single-file greps). A filename-tail brace (`mine-{algorithm,design}-2026-07.md`) worked. When an accept criterion is "grep over these N named files", grep each file individually (or use `**/SKILL.md`), never a directory-level brace glob — an empty result is indistinguishable from "glob matched nothing".
- **Freezing dev-repo research into shipped references: point provenance at the pool entry BY NAME, never its path, and genericize repo-specific rationale.** The shared self-containment gate greps for `docs/kb/research`, `D:\src`, `Omnishelf`, `omnivision-ai-sdk`, `docs/specs/adhoc-` — so a provenance line like "distilled from `docs/kb/research/canonical-algorithm-catalog.md`" trips its own gate; write the pool-entry *name* (`canonical-algorithm-catalog`) instead. Genericize SDK-specific rationale (calib3d `CMakeLists:40,44`, `planogramCompilance`, PLN-1 ids) to stack-neutral phrasing while keeping generic library names (OpenCV/Eigen/STL) which the gate does not forbid.
- **The evaluate-skill Judgment Gate earns its keep even on your own freshly-authored skill — don't rubber-stamp.** Running the rubric adversarially against my own mine-design SKILL.md surfaced a real defect (a dead `### D5 — (reserved …)` contract placeholder I had left mid-authoring). Self-authored + lint-clean + confident is exactly when the artifact-axis gate (job fitness / legibility, which lint structurally cannot see) catches what the author's own eye skips.
- **Fixing a dead pointer: author the content INTO the single-source owner and leave the pointer as-is — never resolve it by restating the content in the pointing skill.** FD1's dead `chunked-writes` pointer had two fixes on the table: (a) author the discipline into the family core §Execution topology the pointer already targets, or (b) inline the one-line discipline in `mine-design/SKILL.md` and drop the pointer. Option (a) is the AP3-correct one — the discipline is family-wide (any large-output member hits the ~64k ceiling), so it belongs in the shared core once and every sibling keeps pointing at it; (b) would have forked a single-source rule into a per-skill copy. When a directive offers both, pick the single-source home unless the content is genuinely skill-private.
- **When adding an absence ACTION to a precondition, patch both the terse pipeline-block cell AND the binding contract (D1), keeping them consistent.** FD2's Stage-0 halt lives in the D1 BR-registry bullet (full sentence + rationale) but the reviewer flagged the *pipeline diagram* as the silent surface — so the diagram cell got a compact echo of the same action. A contract-only fix would have left the flagged diagram still silent; a diagram-only fix would have under-specified the rationale. Two coordinated small touches, same action, matched wording.

## Reviewer Lessons

- **Reviewing prose/skill artifacts: resolve every cross-reference the new text makes, not just the acceptance greps the plan names.** The plan's accept greps were all green fresh, yet the one real defect (mine-design's chunked-writes pointer at a family-core section that doesn't exist) was a *pointer whose target is absent* — a class no positive accept grep catches, because the plan's greps test presence of required content, never resolution of referenced content. For a pointer-heavy artifact (heavy-archetype skill citing a shared core), the review's gap-analysis pass should be "grep the referent for each pointer" — one grep per pointer, cheap and decisive.
- **A plan instruction can presuppose a referent ("X via family-core pointer") that nobody verified exists — check the referent at plan time or at review time.** Plan step 4 directed the chunked-writes discipline "via family-core pointer"; the core never carried it; the developer faithfully authored the dead pointer and the in-context Judgment Gate (which claimed "every core citation resolves live") missed it. Design-origin defects in prose ship exactly this way: an instruction whose premise no gate re-checks. The in-context (non-spawned) evaluate-skill run was flagged as a method-fidelity deviation — this is the concrete fidelity cost.
- **A prose-fix re-review needs the same "resolve the fix, then sweep the neighborhood" discipline as a code fix.** Cycle-1 re-review of FD1/FD2 verified each fix at its exact site (pointer resolves, halt-action present at both the diagram cell and the D1 bullet) AND re-ran skill-lint across the whole 5-member mine family, not just the two touched files — a shared-core edit (FD1 touches `mine-family-core.md`) is exactly the kind of change that can silently regress an unrelated sibling's line-number-sensitive cross-ref. Scoping the re-review to "the two touched files + every sibling that cites the touched section" caught nothing broken here, but the check was cheap (one extra lint invocation) against the cost of a missed regression.
- **An unscoped full-tree `git status` on every fix cycle catches drift a scoped diff never will — run it even when the coordinator names an exact file list.** Cycle 2 was scoped to 5 named files; a scoped scope-check on those 5 passed clean. A *separate*, unscoped `git status --porcelain` on the whole tree (run anyway, as the regression-sweep habit) surfaced two unrelated `docs/research/*.md` deletions and a new untracked `summary.md` that appeared between Cycle 1 and Cycle 2 — neither named by the coordinator, neither explained in implementation.md, and (per the "recheck branch under concurrent run" project lesson) exactly the shape of either concurrent-session contamination in a shared working tree or an undocumented out-of-band edit. Filed as a non-blocking Open Question rather than asserting a cause (the fact is ~95% certain via direct git evidence; the *cause* is unknown, correlation-only) — but it would have gone completely undetected if the review had trusted the coordinator's file list as the sweep boundary. Run the full-tree status check every cycle regardless of how narrowly the fix was scoped.
