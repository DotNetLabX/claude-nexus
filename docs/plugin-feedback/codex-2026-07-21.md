## Job registry is per-runtime — coordinator status checks false-negative on subagent-dispatched jobs
- **Suggested target:** codex → `codex-rescue` / `codex-companion.mjs` status surface
- **Action:** update
- **Evidence:** adhoc-MineSkillAuthoring, recurrence 1x (measured end-to-end: dispatch ack →
  coordinator-side `codex-companion.mjs status` = "No jobs recorded yet" → job was live in the
  rescue agent's own runtime and wrote `review-codex.md` ~35 min after dispatch — a real NO-GO
  with 5 confirmed findings)
- **Lesson:** the job registry is scoped to the runtime that dispatched the job, so a
  coordinator-side status check cannot see a subagent-runtime job and reads as "no job" — a false
  hollow-ack diagnosis. Either persist job records to a location all runtimes share, or make the
  status command state its visibility scope ("jobs dispatched from THIS runtime only") so a
  coordinator doesn't treat an empty registry as proof of a no-show.

## Dispatch ack carries no ETA or liveness signal, and the forwarder cannot self-poll
- **Suggested target:** codex → `codex-rescue` dispatch ack contract
- **Action:** update
- **Evidence:** adhoc-MineSkillAuthoring, recurrence 1x
- **Lesson:** the ack ("Task started as task-{id}") carries no ETA and no liveness channel, and the
  rescue agent's operating rules forbid it from status-polling its own dispatch — so the
  coordinator has no reliable liveness signal at all between ack and file-landing. Add an expected
  time-to-first-write (or a heartbeat/progress file the coordinator can poll) to the ack contract.
