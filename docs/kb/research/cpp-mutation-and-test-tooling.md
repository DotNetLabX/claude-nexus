# cpp-mutation-and-test-tooling

## What C++ tooling fills the mine-verify-cover adapter capabilities (mutation tool, evidence indexer, test framework), and which is the Stryker.NET equivalent?

**Verdict:** mull (v0.34.0, LLVM 13–22) is the closest Stryker.NET equivalent for C++ mutation testing; libclang (Python bindings) is the lowest-friction evidence indexer; GoogleTest + RapidCheck is the recommended test-generation target framework.
**Evidence tier:** read-docs
**As-of:** 2026-06-24
**Validity scope:** LLVM 18.x–22.x; mull 0.34.x; dextool v5.3; GoogleTest 1.x; Catch2 v3; libclang Python bindings (clang.cindex)
**Status:** current
**Reconfirm trigger:** mull LLVM-major version bump; dextool v6 release; any mull JSON schema change breaking the mutation-testing-elements contract
**Corroboration:** high-stakes (Q1 mutation-tool verdict) — 2 sources agree (mull official docs [1–4] + independent Chalmers/Zenseact CI study [5]); second independent source agreed.

## Verdict
For a config-driven, machine-readable-report, CI-drivable C++ mutation harness, **mull is the closest practical Stryker.NET equivalent**: YAML-configured, emits JSON (mutation-testing-elements schema), SQLite, and SARIF, runs fully headlessly via CLI flags, supports LLVM 13–22 on Linux and macOS, and reached v0.34.0 in May 2026 with active monthly releases. Dextool mutate (v5.3) is the stronger choice only if git-diff-scoped mutation or a distributed node setup is required, at the cost of a substantially heavier build (D toolchain, ~16 GB RAM, Linux-only). For evidence indexing, libclang Python bindings give the easiest path to function/class body extraction from a `compile_commands.json`. GoogleTest + `gtest_discover_tests` + RapidCheck is the recommended test target: first-class CTest integration, GMock bundled, and RapidCheck provides FsCheck-equivalent property-based generation.

## Finding
- mull v0.34.0 was released 2026-05-12, with v0.33.0 (2026-04-11), v0.32.0 (2026-04-05 — SARIF reporter added), v0.31.0 (2026-03-25 — LLVM 21 & 22 support), and v0.28.0 (2026-01-27 — LLVM 20 support), demonstrating monthly cadence active into mid-2026 [1].
- mull supports LLVM 13 through 22; prebuilt packages exist per LLVM major (e.g. `mull-13`, `mull-15`, `mull-18`); users pin to a specific LLVM major at install time but the project is not locked to one [2].
- mull integrates with CMake via `compilationDatabasePath` in its YAML config (`mull.yml`), pointing to `compile_commands.json` generated by `-DCMAKE_EXPORT_COMPILE_COMMANDS=ON`; it operates on LLVM IR bitcode injected at compiler-plugin time, with no source rewrite [3].
- mull is fully headless: all options are expressible in `mull.yml` and overridable via CLI flags (`--exclude-path`, `--include-path`, `--workers`, `--quiet`, `--silent`, `--timeout`, etc.) [3].
- mull emits four machine-readable outputs: IDE/console warnings, a SQLite database, JSON conforming to the mutation-testing-elements schema, and (from v0.32.0) SARIF for GitHub Code Scanning [4].
- mull does not officially support Windows; WSL is reported to work but is not an official target. macOS (Homebrew since v0.30.0) and Linux (Ubuntu 22.04/24.04, Debian, Red Hat) are the supported platforms [2].
- mull includes incremental mutation testing via `gitDiffRef` + `gitProjectRoot` config keys, but an independent study found this feature unreliable/removed in a recent update and reliable only in dextool at the time of the study [5].
- dextool mutate v5.3 ("Chert") is the latest release (2025-09); v5.2.0 ("Dolomite") was 2025-08; the project is actively maintained with LLVM 14–20 support (ImportC bindings replace hand-written D bindings, easing future LLVM upgrades) [6].
- dextool mutate uses Clang AST for source-level textual mutation (not IR-level); it requires LLVM ≥ 14.0 and a D-language compiler (ldc for release builds); the build consumes ~9 GB RAM (~16 GB comfortable) [7].
- dextool is the only C++ mutation tool that reliably mutates only changed lines from a `git diff` in CI workflows, confirmed by both the Zenseact/Chalmers case study and the dextool documentation [5].
- dextool stores mutant metadata in SQLite; JSON and HTML report formats are available, and a distributed node setup is supported via the shared SQLite database [8].
- dextool is Linux-only — all build instructions and llvm-config assumptions target Ubuntu/Fedora; no Windows or macOS support documentation exists [7].
- The independent Chalmers/Zenseact CI study (academic, peer-reviewed) evaluated five C++ mutation tools — dextool, mull, MuCPP, Mutate++, CCMutator — and concluded only dextool and mull are suitable for CI; Mutate++ requires a GUI, MuCPP and CCMutator had severe LLVM version-compatibility problems [5].
- libclang provides a stable C API with Python bindings (`clang.cindex`) for AST traversal; it reads `compile_commands.json` directly; function bodies are accessible via `CursorKind.FUNCTION_DECL` / `CXX_METHOD` cursors with `clang_Location_isFromMainFile()` for main-file filtering [9].
- LibTooling offers full AST access and the ASTMatcher DSL but requires building a C++ tool against an in-tree or installed LLVM toolchain — higher setup cost; preferred when macro expansion or template precision is required [10].
- clangd is an LSP server communicating over JSON-RPC; it cannot be used as a programmatic extraction library — suitable only for editor integration, not automated function-body slicing [11].
- GoogleTest ships with GMock and `gtest_discover_tests()` (CMake `GoogleTest` module, since CMake 3.10); it generates a separate CTest test per TEST case (including parameterized instantiations) without requiring CMake re-runs on test additions [12].
- Catch2 v3 provides `catch_discover_tests()` with the same post-build enumeration as GoogleTest; test names are free-form strings (not required to be valid C++ identifiers), which is more readable [13].
- doctest remains header-only (single file) with the fastest compile times; `doctest_discover_tests()` exists in CMake extras; it lacks GMock and has the smallest ecosystem [no source found].
- RapidCheck integrates with GoogleTest via `RC_GTEST_PROP` macros and with Catch2 via `rc::prop` inside `TEST_CASE`; it is added via CMake `add_subdirectory`/`FetchContent` and linked as `rapidcheck::rapidcheck` [14].

## Fix
- Wire mull as the mutation gate: compile with Clang (`-fpass-plugin=mull-ir-frontend`), point `compilationDatabasePath` in `mull.yml` at the CMake-generated `compile_commands.json`, set `reporters: [SQLite, Elements]`, and parse the SQLite DB or JSON for the kill-score floor [3].
- Install the LLVM-version-pinned mull package (e.g. `mull-18`) matching the project's Clang version — the generic `mull` package is no longer updated [1].
- For the evidence indexer, use `clang.cindex` (`pip install libclang`) against `compile_commands.json`; iterate `CursorKind.FUNCTION_DECL`/`CXX_METHOD` cursors and filter with `clang_Location_isFromMainFile()` to get per-function source extents for LLM slicing [9].
- Add GoogleTest via CMake `FetchContent`, enable `gtest_discover_tests(target)`, link `rapidcheck::rapidcheck` for property-based generation, and capture CI artifacts with `ctest --output-junit results.xml` [12].
- If git-diff-scoped mutation is required, plan for dextool on a Linux CI node rather than mull — mull's `gitDiffRef` has known reliability issues [5].

## Alternatives
- dextool mutate v5.3 — richer CI integration (git-diff mutation, distributed nodes, JSON + HTML), but Linux-only, needs the D toolchain, ~9 GB compile RAM, and uses Clang-AST source rewriting rather than IR injection; better for safety-critical/industrial contexts (used at Saab Aeronautics) [5][6].
- MuCPP — cannot skip uncovered mutants (CI-impractical); LLVM version incompatibilities require Docker isolation [5].
- Mutate++ — GUI-required workflow, not automatable [5].
- CCMutator — LLVM version incompatibilities; not CI-recommended by the Chalmers study [5].
- universalmutator — language-agnostic regex-based source mutator; no C++ AST awareness, high equivalent-mutant rate; no evidence of active 2025–2026 C++ production use [no source found].
- LibTooling (for indexing) — full AST fidelity including macros and templates, but requires building a C++ tool against LLVM headers (high setup); prefer for macro-heavy code where libclang cursors misbehave [10].
- Catch2 v3 (test framework) — free-form test names, simpler setup, `catch_discover_tests` parity with gtest, but loses GMock; reasonable if the harness needs no mock injection [13].

## Caveat
- mull requires the project to compile with Clang/LLVM — it does not work on GCC-compiled binaries; a GCC-locked project conflicts [2].
- mull's SARIF output (v0.32.0+) is separate from the mutation-testing-elements JSON; a kill-score parser must target the JSON or SQLite, not SARIF [4].
- libclang Python bindings have known deficiencies with macros and complex templates — `clang_Location_isFromMainFile()` does not suppress all transitive header content in every case [10].
- dextool's build requirement (D toolchain + ldc + ~16 GB RAM) makes it impractical in a standard Docker image without a pre-built binary; no official binary distribution comparable to mull's Cloudsmith packages was found [7].
- mull incremental `gitDiffRef` testing may not be reliable (reportedly removed/broken as of the Chalmers study); verify on the exact mull version before relying on it [5].
- RapidCheck is not under active major development (last major release 2017); it works with modern GoogleTest but its generator/shrinking ecosystem is smaller than FsCheck (.NET) or Hypothesis (Python) [14].

## Fallback
- If mull cannot compile the target (e.g. GCC-only CI), fall back to dextool mutate on a dedicated Linux runner, accepting the D-toolchain build overhead and Linux-only constraint [5].
- If mull's JSON schema changes, switch the kill-score parser to query the co-emitted SQLite DB directly — less likely to change than the JSON consumer format [4].
- If libclang cursors fail on macro-heavy code, shell out to `clang++ -Xclang -ast-dump=json` and parse the JSON tree — no library dependency, but slower and verbose [9].
- If RapidCheck is insufficient for a property, use a dedicated C++ fuzzer (libFuzzer) for coverage-guided generation outside the unit-test harness [no source found].

**Immediate next action:** add a mull mutation step to the C++ adapter — install the LLVM-version-pinned package, add `mull.yml` (`compilationDatabasePath`, `reporters: [Elements, SQLite]`, `workers: N`), and parse `mull-report.json` (or the SQLite DB) for the kill-score floor gate.
