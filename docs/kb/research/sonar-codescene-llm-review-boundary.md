# SonarQube & CodeScene Coverage Boundary for an LLM PR Reviewer

## Which rule/check classes do SonarQube and CodeScene cover deterministically today, and which review concerns fall outside what these static analyzers can check?

**Verdict:** SonarQube and CodeScene deterministically cover syntax/pattern-level bugs, known vulnerability patterns, computable complexity/size/duplication metrics, and (with manual YAML/JSON configuration) declared layering/dependency constraints; CodeScene additionally covers git-history-derived hotspot, coupling, and knowledge-distribution metrics. Both vendors' own docs and independent sources agree that intent-based judgment — semantic naming correctness, unwritten/project-specific conventions, "does this code serve its architectural purpose" reasoning, and business-logic/domain-fit correctness — sits structurally outside what either tool's rule engine evaluates, even after their 2025-2026 AI additions (AI CodeFix/Assurance, CodeScene ACE), which still only fix or auto-refactor findings the deterministic engine already flagged.
**Evidence tier:** read-docs
**As-of:** 2026-07-09
**Validity scope:** SonarQube Server (commercial editions, 2025.x–2026.1 LTA) and SonarQube Cloud (Team/Enterprise plans) as of mid-2026; CodeScene Enterprise/Cloud docs (7.x line) as of mid-2026. Holds for the "Standard Experience" and MQR rule models; the newer cross-language Architecture-as-Code feature is explicitly beta/in-flux (old Java-only architecture module is deprecated, removal pending Jan 2026).
**Status:** current
**Reconfirm trigger:** GA release of SonarQube's new multi-language Architecture feature (currently beta on SonarQube Cloud) or its arrival on SonarQube Server; any SonarQube/CodeScene feature that adds LLM-based semantic/intent review rather than fix-generation for already-flagged findings; CodeScene ACE expanding auto-refactor beyond the current five code-smell types; major version bump of either tool's rule/quality model.
**Corroboration:** 17 sources total (10 vendor docs, 7 independent). The core boundary verdict ("static analyzers evaluate code in isolation without architectural/intent understanding; conventions and intent require human or LLM judgment") is corroborated independently by at least three unrelated sources: an enterprise AI-code-review vendor analysis [13], a static-analysis-vendor's own blog on what their tool cannot find [14], and an unrelated engineering blog describing why they built an LLM reviewer specifically to fill this gap [16].

## Verdict
- Deterministic coverage today = pattern-matched bugs/vulnerabilities, computable metrics (complexity, size, duplication), naming-format regex rules, and manually-declared architecture/layering constraints (Sonar) plus git-history-derived hotspot/coupling/knowledge metrics (CodeScene) [1][2][9][11].
- Both tools' 2025-2026 "AI" features (AI CodeFix/AI Code Assurance, CodeScene ACE) generate or apply fixes for issues the deterministic engine already detected — they do not add independent semantic/intent judgment [6][7][10].
- Semantic naming correctness, unwritten/team-specific conventions, "does this code serve its intended purpose" reasoning, and business-logic/domain correctness are outside both tools' structural reach and are the areas an LLM PR reviewer should own [12][13][14][16].

## Finding
- **SonarQube rule taxonomy:** SonarQube's Standard Experience mode has four rule types — Bugs (reliability), Code Smells (maintainability), Vulnerabilities (security), and Security Hotspots (security-sensitive code needing manual review); MQR mode reorganizes the same underlying rules around three software qualities (security, reliability, maintainability) [1].
- The platform ships 6,500+ rules across 35+ languages, and rule assignment follows a deterministic decision tree (exploitable → vulnerability; security-sensitive → hotspot; demonstrably wrong → bug; otherwise → code smell) [1].
- SonarQube is gradually retiring the standalone "Security Hotspot" type in favor of folding it into vulnerabilities/security issues, but hotspots today are explicitly non-deterministic findings: the docs state a hotspot flags security-sensitive code without asserting the app is actually impacted, and it is "up to the developer to review" and decide if a fix is needed given context like budget and threat model — i.e. Sonar's own docs concede this class requires human judgment, not a deterministic verdict [11][1].
- Naming-convention rules (e.g., Java S100/S116 "names should comply with a naming convention") are deterministic regex-pattern checks — they validate lexical form (casing, prefix/suffix patterns), not semantic appropriateness of a name [no source found]
- Complexity metrics (cyclomatic complexity, SonarSource's own Cognitive Complexity formula) and duplication detection are deterministic, formula-based computations over the AST/token stream, not learned or LLM-driven [1][14].

- **SonarQube architecture/layering support:** SonarQube does support architecture/dependency rules, but only via explicit manual configuration: teams "Discover" the actual dependency graph automatically, then must "Formalize" intended architecture by declaring it as code in a YAML/JSON file (or, in the newer UI, a visual "intended architecture editor") before Sonar can "Prioritize" and "Fix" (raise issues for) deviations — it does not infer what the *right* architecture should be [3][5].
- There are two separate architecture capabilities in flight: (a) the legacy Design & Architecture module bundled in all commercial SonarQube Server editions, historically Java-only, whose cycle-detection/architecture-as-code parts are deprecated with removal pending January 2026 [4]; and (b) a newer cross-language "Architecture" feature (Java, JavaScript, TypeScript, Python, C#) that is beta on SonarQube Cloud and not yet ported to SonarQube Server [3][5].
- Even where configured, architecture constraints only catch deviations from a rule the team explicitly wrote (e.g., forbidden Presentation→Database import, circular dependency) — they do not evaluate whether an undeclared, ad hoc, repo-specific pattern (e.g., "this service always validates input in a specific helper before touching the DB") is being followed, since no such pattern is ever declared as a formal constraint [3][5].

- **SonarQube's AI features:** AI CodeFix uses an LLM (SonarSource-hosted or a customer's own/self-hosted model) purely to generate a suggested fix for an issue the deterministic rule engine has already flagged; it is gated to Enterprise/Data Center editions (Server) or Team/Enterprise plans (Cloud) [6].
- AI Code Assurance is a separate, CodeFix-independent capability that labels AI-generated code, applies custom quality gates to such code, and issues monitoring badges — it is framed by Sonar as designed to catch "unique risks... that may be overlooked by traditional static analysis" in AI-authored code, but the documentation describes it as gating/labeling plus the existing rule engine, not a new semantic-review capability [7].

- **CodeScene coverage:** CodeScene's Code Health score (1-10) is built from 25-30 factors per language, mixing purely deterministic structural metrics (cyclomatic complexity, lines of code, nesting depth, LCOM4 cohesion) with heuristic pattern-detectors that combine multiple structural signals under an interpretive label (Brain Method/Class, Bumpy Road, Primitive Obsession, DRY violations detected via change-history correlation) [9].
- CodeScene's hotspot analysis is explicitly algorithmic/statistical, not LLM-based: it starts from git change-frequency data cross-referenced with low Code Health to prioritize risk, and Change Coupling surfaces files that historically change together to reveal hidden/architectural coupling — all derived from repository history mining, not code semantics [9].
- CodeScene's "social"/behavioral layer (Knowledge Maps, Bus Factor, Coordination/Fragmentation) computes ownership and fragmentation percentages from commit authorship data, not from any judgment about code meaning [no source found]
- The Code Health documentation contains no mention of LLM or AI involvement in computing the score itself — it is rule-based/statistical over code structure and VCS history [9].
- CodeScene ACE is an AI-based auto-refactor add-on, but its scope is narrow and enumerated: it currently auto-refactors only five code-smell types (Large Method, Deep Nested Logic, Bumpy Road, Complex Conditional, Complex Method), fact-checking the LLM's proposed refactor against the original code's behavior before offering it — again, fixing what the deterministic Code Health engine already flagged, not adding new judgment categories [10].

- **The boundary — what falls outside static analysis structurally:** Multiple independent sources converge on the same structural gap: static analyzers evaluate code "in isolation, without understanding its architectural purpose," so they can flag a defensive null check as redundant when it is actually required public-API protection — a distinction that needs contextual/intent understanding [13].
- The same source states static tools structurally cannot assess cross-service breaking changes, downstream impact across a distributed system, or distinguish compliance-mandated complexity (audit logging, access control) from "unnecessary" complexity — these require reasoning beyond single-file/module inspection [13].
- A static-analysis-tool vendor's own blog states plainly that determining code correctness requires knowing "what the author of the code intended to write" and domain knowledge (their example: `sin()` vs `cos()` is syntactically valid but semantically wrong) — something only a human (or an entity with equivalent domain understanding) can judge, not a pattern-matcher [14].
- An independent engineering team that built an LLM code-review tool specifically to close this gap reports that "many of our standards depend on intent, not just structure" — e.g., a function returning a large dict might represent structured data (should be a dataclass) or a computed result (fine as a dict), and "a linter can't reliably distinguish those cases, but a human reviewer can"; this was their explicit reason for reaching for an LLM rather than extending their linter [16].
- A peer-reviewed survey of static analysis tool limitations documents a further structural reason beyond semantics: tools deliberately trade soundness for precision/performance (the underlying data/control-flow problems are undecidable at scale — related to the halting problem), and these unsound trade-offs are often undocumented, producing "misplaced confidence" — reinforcing that even the deterministic layer has known, admitted blind spots on top of the semantic ones [17][14].
- A third independent source (SAST-critical CI/CD blog) reports ~5% false-positive rates at scale, arguing SAST findings "ultimately rel[y] only on the assessment of a human being" to adjudicate — i.e., even inside its nominal deterministic domain, the tool's output still requires human/LLM triage judgment before it's actionable [15].

## Fix
- Exclude from the LLM reviewer's charter: anything expressible as a deterministic pattern/regex/AST rule (naming lexical format, known-bad API usage, cyclomatic/cognitive complexity thresholds, duplication, declared circular-dependency/layering violations) — SonarQube/CodeScene already own this and do it exhaustively and cheaply [1][9].
- Exclude: git-history-derived metrics (hotspots, change coupling, bus factor, fragmentation) — this is CodeScene's specific, mature domain and re-deriving it via LLM would be redundant and less reliable than the statistical approach [9].
- Exclude: generic security vulnerability pattern matching (SQL injection via string concatenation, hardcoded secrets, known-CWE patterns) covered by SonarQube's Vulnerability rules — but note Security Hotspots are only *flagged*, not resolved, by Sonar, so an LLM reviewer could legitimately add contextual judgment on top of a hotspot flag Sonar has already surfaced [11].
- Include: judgment of whether a name is *semantically* correct/misleading for what it does (not just lexically formatted) [14][16].
- Include: whether code conforms to unwritten/project-specific conventions and patterns that were never formalized as a Sonar architecture-as-code YAML rule or a custom lint rule — because Sonar's architecture feature only catches deviations from constraints a human explicitly declared, and undeclared conventions are invisible to it [3][5].
- Include: whether a change serves its stated architectural/business intent (e.g., is a null check defensive-necessary or dead code; is added complexity compliance-mandated or accidental) — explicitly named by an independent source as the kind of judgment static tools cannot make [13].
- Include: cross-cutting/cross-service intent and domain-fit correctness that single-file/module-scoped static analysis structurally cannot see [13][17].

## Alternatives
- Draw the boundary purely by "tool absence": scope the LLM reviewer only to what's missing when a specific repo has no Sonar/CodeScene configured at all, rather than by category — simpler to operate but risks scope creep into deterministic territory already covered elsewhere [no source found]
- Draw the boundary by "confidence tier": let the LLM reviewer also comment on Sonar Security Hotspots and CodeScene heuristic factors (Brain Method, Primitive Obsession) since vendor docs concede these already need human judgment [9][11].
- Draw the boundary by "fix vs. judge": let AI CodeFix/ACE own mechanical fixes to already-flagged issues, and scope the LLM PR reviewer strictly to net-new judgment calls the deterministic engines never flag at all — closest match to how the vendors themselves already split AI features (fix-generation) from rule-detection (deterministic engine) [6][10].

## Caveat
- SonarQube's newer, cross-language Architecture feature is beta (SonarQube Cloud only, Java/JS/TS/Python/C#) and the older Java-only architecture module is being deprecated with removal pending January 2026 — the "does Sonar do architecture at all" answer is actively shifting and should be reconfirmed before relying on it for scope decisions [3][4][5].
- Edition/plan gating matters: AI CodeFix requires Enterprise/Data Center (Server) or Team/Enterprise (Cloud); architecture constraints require a commercial edition; a project on SonarQube Community Edition or CodeScene's lower tiers may have none of the above capabilities, effectively widening the LLM reviewer's proper scope on that project [6][4].
- Two claims above (regex mechanics of SonarQube's S100/S116 naming rules; CodeScene's Bus Factor/Fragmentation computation details) are corroborated only via search-result synthesis rather than a directly fetched primary doc page — flagged inline as [no source found], though directionally consistent with the fetched Code Health documentation [9].
- "Deterministic" for CodeScene heuristic factors (Brain Method, Bumpy Road, Primitive Obsession) is a matter of degree: these combine multiple deterministic sub-metrics under a named pattern, so they are rule-based and reproducible but were empirically tuned/thresholded, not literally a single formula — the entry's Corroboration and Fix sections treat them as "in scope for CodeScene" regardless of this nuance [9].
- Sources are a mix of vendor documentation (which has incentive to overstate capability) and independent/critical sources (which may have incentive to overstate limitations to sell an alternative, e.g., Augment Code's own AI review product, PVS-Studio's own static analyzer, Code Intelligence's own fuzzing product) — the boundary verdict is corroborated across vendors with different commercial interests (Sonar/CodeScene docs, an AI-review vendor, a static-analysis vendor, an unrelated engineering team with no tool to sell), which mitigates but does not eliminate this bias.

## Fallback
- If a project has no SonarQube/CodeScene (or only Community Edition with no architecture/AI features), the LLM PR reviewer's charter should temporarily widen to also flag the deterministic-tier issues those tools would otherwise catch (obvious bugs, glaring complexity, duplication) until such tooling is added, since nothing else in the pipeline is covering that ground [1][9].
- Where only partial capability exists (e.g., SonarQube without the Enterprise architecture module, or CodeScene without ACE), the LLM reviewer should absorb exactly the missing tier (e.g., own layering/dependency conformance review if no architecture-as-code file exists) rather than assuming full deterministic coverage is present — verify per-repo which SonarQube edition/CodeScene tier is actually configured before finalizing scope for that repo [no source found]

## Recommendation
Charter the LLM PR reviewer to explicitly exclude everything expressible as a deterministic rule (naming lexical format, complexity/size thresholds, duplication, known vulnerability patterns, and any layering/dependency rule already declared in a Sonar architecture-as-code file) and everything derivable purely from git history (hotspots, change coupling, bus factor/knowledge fragmentation) — these are SonarQube's and CodeScene's proven, mature domain [1][9]. Have it own instead: semantic correctness of names/intent versus their syntactic form, conformance to unwritten/undeclared project-specific conventions that never made it into a formal Sonar constraint or lint rule, judgment calls on whether a pattern (defensive check, added complexity, cross-service change) serves its architectural/business purpose, and adjudication of Security Hotspots that Sonar flags but explicitly leaves to human review [3][5][11][13][16]. Re-verify this boundary if SonarQube's Architecture feature reaches GA on Server, or if either vendor ships a feature that performs intent-level semantic review rather than fix-generation on top of existing deterministic findings.
